## Gaps surfaced

### GAP #1 (CRITICAL — compiler) — `BeliefsAccessor` has no CG-lower path

`crates/dsl_compiler/src/cg/lower/expr.rs:817-820`:

```rust
IrExpr::BeliefsAccessor { .. } => Err(LoweringError::UnsupportedAstNode {
    ast_label: "BeliefsAccessor",
    span,
}),
```

Companion arms at lines 821-828 reject `BeliefsConfidence` and
`BeliefsView` identically — the entire family of belief-read IR
nodes dies at the AST → CG bridge.

**Where the IR comes from.** Parse + resolve work end-to-end:
- Parser (`crates/dsl_ast/src/parser.rs:2071-2156`) accepts
  `beliefs(observer).about(target).<field>` /
  `.confidence(target)` / `.<view>(_)` and produces
  `ExprKind::BeliefsAccessor` / `BeliefsConfidence` / `BeliefsView`.
- Resolver (`crates/dsl_ast/src/resolve.rs:2132-2162`) validates the
  `<field>` against the 6-element `BELIEF_FIELDS` allowlist
  (`last_known_pos`, `last_known_hp`, `last_known_max_hp`,
  `last_known_creature_type`, `last_updated_tick`, `confidence`) and
  emits `IrExpr::BeliefsAccessor`. Errors otherwise with a typed
  `ResolveError::UnknownBeliefField`.
- The IR shape is preserved through the CG-lower's recursive walker
  (`expr.rs:2134`) — it's just the leaf-lowering that rejects.

**Likely fix surface (next-step task).** Add a real lowering arm:

```rust
IrExpr::BeliefsAccessor { observer, target, field } => {
    // Lower observer + target as AgentId expressions
    let observer_id = lower_expr(observer, ctx)?;
    let target_id   = lower_expr(target, ctx)?;
    // Field selects a slot in the per-(obs, tgt) belief tuple
    let slot = belief_field_slot(field)?;  // u32 offset into BeliefState SoA
    let handle = DataHandle::BeliefField {
        slot,
        observer: observer_id,
        target: target_id,
    };
    add(ctx, CgExpr::Read(handle), span)
}
```

Plus (a) add a `DataHandle::BeliefField` variant to
`crates/dsl_compiler/src/cg/data_handle.rs`, (b) wire it through
the WGSL emit in `cg/emit/expr.rs` to index the belief SoA buffer
(see Gap #3), (c) extend the per-kernel binding allocator in
`cg/emit/kernel.rs` to declare the belief storage binding when the
kernel body references a `BeliefField`.

The 6 fields have heterogeneous types (`Vec3`, `f32`, `f32`,
`CreatureType` enum, `u32`, `f32`); the SoA layout will need one
buffer per field unless they're packed into a single struct-of-bytes
buffer with field-offset reads.

### GAP #2 (CRITICAL — compiler) — `theory_of_mind.believes_knows()` has no CG-lower path

`crates/dsl_compiler/src/cg/lower/expr.rs:2649-2668` (the namespace-
call fallback):

```rust
_ => {
    if let Some(def) = ctx.namespace_registry.namespaces.get(&ns)
        .and_then(|nd| nd.methods.get(method))
    {
        return lower_registered_namespace_call(ns, method, args, span, ctx, def.clone());
    }
    Err(LoweringError::UnsupportedNamespaceCall {
        ns,
        method: method.to_string(),
        span,
    })
}
```

`theory_of_mind` is registered in the resolver's namespace table
(`crates/dsl_ast/src/resolve.rs:145`) and its three methods are
sigged in `method_sig` (`resolve.rs:505-511`):

```rust
(NamespaceId::TheoryOfMind, "believes_knows") => Some((3, IrType::Bool)),
(NamespaceId::TheoryOfMind, "can_deceive")    => Some((3, IrType::Bool)),
(NamespaceId::TheoryOfMind, "is_surprised_by") => Some((3, IrType::Bool)),
```

But there is NO entry in the CG-side `namespace_registry` (search
`grep -rn 'TheoryOfMind' crates/dsl_compiler/src/cg/` returns
nothing) — so the lookup `nd.methods.get("believes_knows")` returns
`None` and the call falls through to `UnsupportedNamespaceCall`.

The spec (`docs/spec/dsl.md:1070`) describes the semantic: a "bit
read against the 32-bit `Relationship.believed_knowledge` domain
bitset". This is the SHAPE that matches the task prompt's
"`agent.beliefs(other, fact_id) -> bool`" — but it's a per-PAIR
relationship structure, NOT a per-agent self-bitset. The third arg
is a `domain` (e.g. Combat, Trade, Family, ...) drawn from a 32-bit
enum; today no `DomainId` enum exists in the IR (see comment at
`resolve.rs:497`).

**Likely fix surface (next-step task).** Two-step:
1. Land `DomainId` (or `FactId`) as a real `IrType::Named` — makes
   the third arg type meaningful instead of `Unknown`.
2. Register `(TheoryOfMind, believes_knows)` in the CG namespace
   registry with a custom lowering hook (similar to `RngPurpose` —
   the registry path does shape mapping, not just sig validation).
   The hook lowers to a `CgExpr::Read(DataHandle::BeliefBit {
   observer, subject, domain_bit })` — a `bool` read against the
   per-(observer, subject) bitset SoA.

### GAP #3 (CRITICAL — runtime/architecture) — no per-(observer, target) belief storage SoA exists on the GPU

The closest existing infrastructure:

- CPU-side: `SimState.cold_beliefs: Vec<BoundedMap<AgentId,
  BeliefState, 8>>` (`crates/engine/src/state/mod.rs:138`),
  feature-gated on `theory-of-mind` (Cargo.toml:55). One
  `BeliefState` (6 fields, ~36 B) per slot, capacity 8 per observer.
  Hand-written; not GPU-resident; not bound through any kernel.
- GPU-side: nothing. There's no `belief_storage_*` buffer in any
  emitted kernel binding; no `BoundedMap`-equivalent on the GPU.

Even if Gap #1 closed (`BeliefsAccessor` lowering produces a `Read`),
the runtime would hit a missing-binding error at emit time because
no kernel knows how to allocate or bind a per-(observer, target)
belief buffer.

**Open design question (the spec doesn't pick a side).** The task
prompt says "per-agent bitset SoA" — that's the cheapest, but it's
the `believes_knows` shape, not the `BeliefsAccessor` shape:

| Surface | Storage | Memory at N=2048 |
|---------|---------|------------------|
| `believes_knows(o, s, dom)` | Per-(o, s) `Bitset<32>` — **per-pair** flat: `N²·4 B` | 16 MB |
| `believes_knows(o, s, dom)` | Per-(o, s) bitset stored in `Relationship` rows — but `Relationship` is itself a sparse map | depends on connectivity |
| `BeliefsAccessor(o, t).<field>` | Per-(o, t) `BeliefState` (~36 B) — sparse `BoundedMap<AgentId, ., 8>` per observer | 36 B × 8 × N ≈ 600 KB |
| `BeliefsAccessor(o, t).<field>` | Per-(o, t) flat `BeliefState[N²]` | ~150 MB at N=2048 (probably not viable) |

The CPU-side `BoundedMap<AgentId, BeliefState, 8>` shape suggests
the design intent is sparse, but the GPU doesn't have a native
bounded-map primitive — the existing `engine::pool::BoundedMap` is
heap-allocated and not Pod. A GPU port likely needs:
- A flat per-observer ring of `(target_id: AgentId, slot: u32)`
  pairs (8 entries each) for the lookup index.
- A flat per-(observer, slot) struct-of-buffers for the field data.
- A binary-search probe per `BeliefsAccessor` read (~3 hops at
  cap=8) to find the slot index from the target_id.

This is a meaningful runtime / infrastructure decision — large enough
that landing it in the same task as the lowering would have been
wrong. Punting to follow-up.

### GAP #4 (CRITICAL — compiler/spec) — the spec mismatches itself on belief shape

The spec's two surfaces describe DIFFERENT semantic objects with
DIFFERENT storage:

- `BeliefsAccessor` (parser, resolver, IR — landed): per-(observer,
  target) `BeliefState` tuple — observation snapshot (last-known
  pos / hp / type / tick + confidence). Six named scalar fields.
- `believes_knows` (resolver `method_sig`, dsl.md §7.2): per-
  (observer, subject) bit in the 32-bit `Relationship.
  believed_knowledge` bitset — observer's model of subject's
  knowledge of a domain.

These are NOT redundant — they're complementary layers (Phase 1
"physical snapshot" vs Phase 2 "knowledge model" per
`docs/spec/ability.md:1432`). But the .sim DSL surface treats them
as if they shared a namespace (`beliefs(...).about(...)` looks
syntactically similar to `theory_of_mind.believes_knows(...)`),
which is going to be confusing for fixture authors. The `Relationship`
struct also doesn't exist in `crates/engine` today (audit note in
`docs/spec/state.md:175` — "the `Memory.beliefs: Vec<Belief>` layer
is entirely absent").

**Likely fix surface (separate spec/design task).** Either:
- Decide that `BeliefsAccessor` IS the canonical surface and retire
  the `theory_of_mind.believes_knows()` namespace fn (sigged but
  unused). Land an additional `BELIEF_FIELDS` entry like
  `believes_knows: Bitset<32>` so the .about-style read covers the
  knowledge-bit semantic.
- OR: keep both, land `believes_knows` as a compiler-emitted bit-
  read against a per-(observer, subject) `Relationship` table (lots
  more work — `Relationship` itself isn't allocated yet).

### GAP #5 (compiler — cosmetic) — IR sidecar arms exist but are unreachable

`crates/dsl_compiler/src/cg/lower/expr.rs:2134` (the recursive walker
that copies BeliefsAccessor through structural lowering) and
`crates/dsl_compiler/src/cg/lower/driver.rs:1332` and
`crates/dsl_compiler/src/schema_hash.rs:832` all carry working
implementations for `BeliefsAccessor` / `BeliefsConfidence` /
`BeliefsView`. The schema hash already accounts for these IR nodes
in the deterministic hash — meaning a checkpoint hash collision
would be detected if the gaps closed without touching schema_hash.

This is GOOD — when the lowering gap closes, the schema hash
plumbing is already in place. But the unreachable IR-walker arms
make the surface look "almost-implemented" when in fact the only
thing blocking it is the leaf rejection at `expr.rs:817`.

### GAP #6 (engine — feature flag has no enforced semantics)

`crates/engine/Cargo.toml:55`:

```toml
theory-of-mind = []
```

The flag exists and DOES gate the `cold_beliefs` field (search
`grep -rn 'feature = "theory-of-mind"' crates/engine` shows 6
sites, all in `state/mod.rs` for the field decl/init/clear/getter
shape). But:
- The flag is OFF in the `default = []` set, so `cargo build` (no
  features) leaves `cold_beliefs` absent entirely — even the
  hand-written CPU storage isn't there.
- The DSL fixtures don't enable this feature; the
  `tom_probe_runtime` build doesn't enable it; `sim_app` doesn't.
- When the lowering gaps close, the runtime will need to either (a)
  always-on the GPU belief storage (and drop the feature gate), or
  (b) emit kernels that conditionally compile based on whether the
  fixture's program actually contains a `BeliefsAccessor` IR node.

The cleaner answer (per `docs/CLAUDE.md`'s "rules-as-data"
philosophy) is (b) — but that's a larger refactor of the build
script's shape (build.rs would need to inspect the IR for ToM uses
before deciding which bindings to allocate). Today neither path is
implemented.

## Suggested next steps (separate tasks — sequential, not parallel)

**Decision step (Gap #4):** before any code lands, pick the canonical
surface. Probably "BeliefsAccessor wins, believes_knows retires" —
the BeliefsAccessor surface is more flexible (6 typed fields vs 32
opaque bits) and the CPU storage is already there. Discuss in the
spec / ADR before code commits.

**Implementation order, assuming "BeliefsAccessor wins":**

1. **Land `DataHandle::BeliefField` + the lowering arm** (Gap #1).
   - `crates/dsl_compiler/src/cg/data_handle.rs` — new variant
     `BeliefField { observer: CgExprId, target: CgExprId, slot: u8,
     ty: CgTy }`.
   - `crates/dsl_compiler/src/cg/lower/expr.rs:817` — replace the
     reject arm with the lowering shown in Gap #1.
   - `crates/dsl_compiler/src/cg/emit/expr.rs` — emit
     `belief_storage_<slot>[observer * MAX_TARGETS_PER_OBSERVER +
     bsearch(observer, target)]`.

2. **Land the runtime per-(observer, target) belief storage** (Gap
   #3). Pick "sparse per-observer ring" approach:
   - Per-observer 8-slot index buffer: `belief_target_idx:
     array<u32>` (size = N × 8); slot value = 0xFFFFFFFF if empty.
   - Per-(observer, slot) field buffers: 6 buffers, one per
     `BeliefState` field, all sized N × 8.
   - bsearch helper in WGSL (8 entries → 3 compare-and-branch).
   - Bind through every kernel that references `BeliefField`.

3. **Initialise belief storage from a fixture init hook.** The
   tom_probe_runtime would seed e.g. `belief_storage_confidence[i*8
   + 0] = 1.0` and `belief_target_idx[i*8 + 0] = i` for half the
   slots — proving the GAP-free probe lights up only those slots.

4. **Add `BeliefObserve` cascade physics rule lowering** (separate
   gap, mentioned in `docs/spec/state.md:244`). The `beliefs(o).
   observe(t) with { ... }` statement form parses today
   (`parser.rs:1968-2020`) but its CG-lowering path is unverified
   for this probe (the probe only exercises READS).

5. **Re-run `cargo run -p sim_app --bin tom_probe_app` after each
   step.** Expected progression:
   - After step 1 only: build still succeeds (lowering produces a
     `Read` against missing binding), but emit fails with a
     "missing binding" error. Diagnostic surface improves but no
     observable.
   - After steps 1+2: kernel emits clean; runtime build succeeds;
     `fact_witnesses[i] = 0.0` because storage is uninitialised
     (every confidence read returns 0.0, predicate `> 0.5` false).
   - After steps 1+2+3: `fact_witnesses[i] = 100.0` for the seeded
     half of slots, 0.0 for the rest. **OUTCOME (a) FULL FIRE.**

6. **Lock the FULL FIRE outcome in the
   stress_fixtures_compile test** by flipping the assertions from
   "expected diagnostic" to "expected `physics_CheckBelief` kernel
   in artifacts" + "expected `belief_storage_confidence` binding in
   that kernel's body".

7. **Decide on `believes_knows`.** If retired: remove the
   `method_sig` arms in `resolve.rs:505-511`; replace with
   `Belief.believes_knows(observer, subject) -> Bitset<32>` (or
   similar, a 32-bit field on `BeliefState`). If kept: do the
   `Relationship` allocation work (separate, larger task).
