# duel_25v25 — first SCALE-UP fixture (2026-05-04)

## Outcome

**(a) FULL FIRE** — the 25v25 squad scuffle resolves end-to-end via
spatial body-form physics. Per-tick spatial-hash builds → ScanAndStrike
walks 27-cell neighbourhoods + emits Damaged for every cross-team
neighbour in range → ApplyDamage chronicle decrements HP + sets
alive=false on transition + emits Defeated.

```
Tick   0: Red=25 Blue=25  total_hp(Red)=1250 total_hp(Blue)=1250
Tick   1: Red=25 Blue=25  total_hp(Red)=1125 total_hp(Blue)=1125  ← first volley
Tick  25: Red= 0 Blue= 0  total_hp(Red)=5    total_hp(Blue)=0     ← BATTLE ENDS
```

Final state: mutual annihilation at tick 25. Both squads fully
eliminated, total_damage=55180, total_defeats=745. The OUTCOME (a)
classifier fires: damage > 0, defeats > 0, at least one squad zeroed.

The "draw at tick 25" is a real spatial outcome of the dense initial
layout (5×5 grid per side, x∈[-2,0) for Red and (0,2] for Blue, gap of
0.4 between teams). Every Combatant in the seam zone has 3-5 enemy
neighbours in their 1.5-radius cell window from tick 0; with cooldown=2
and damage=5.0, all 50 die in roughly the same window.

## Closed gaps (inline fixes in this commit)

### Gap M — `ForEachNeighborBody` walker missed implicit `agent_pos` self-cell read

**File**: `crates/dsl_compiler/src/cg/stmt.rs:771-790` (extended).

The WGSL emit template for the body-form spatial walk
(`emit_for_each_neighbor_body` in
`cg/emit/wgsl_body.rs:2202`) reads `agent_pos[agent_id]` to compute
`_self_cell_f` for the 27-cell neighbourhood window — regardless of
whether the body itself references `self.pos`. Pre-fix the
`collect_stmt_dependencies` arm for `CgStmt::ForEachNeighborBody`
surfaced only the GridCells / GridOffsets / GridStarts spatial
storage reads, so the BGL composer omitted the `agent_pos` binding
when no body statement explicitly read it.

`particle_collision_min` happens to read `self.pos` in its
MoveParticle integrator body, so the implicit self-cell read aligned
with the explicit body read and the kernel got `agent_pos` bound by
coincidence. `duel_25v25`'s ScanAndStrike body never reads `self.pos`
(only the spatial walk needs it, and only at the WGSL-template level)
— surfaced as `no definition in scope for identifier: agent_pos` in
the emitted WGSL.

Fix: append a `DataHandle::AgentField { field: Pos, target: Self_ }`
read to the `ForEachNeighborBody` arm in `collect_stmt_dependencies`.
The walker now mirrors the WGSL template's actual read set rather
than only the body's explicit references.

## Inherited deferred gaps (from duel_1v1)

Same gaps the duel_1v1 fixture surfaced, all advisory:

### Gap X — P6 violation false positive for `@phase(post)` chronicle physics

`well_formed::check_well_formed` reports:
```
op#3: P6 violation — physics_rule writes agent field agent.target(#24).hp
op#3: P6 violation — physics_rule writes agent field agent.target(#29).alive
```

ApplyDamage is a chronicle physics rule that legitimately writes per-
agent SoA via `agents.set_hp` / `agents.set_alive`. P6 should
distinguish `@phase(post)` rules from view-fold writes; today it
flags both. Same advisory as duel_1v1.

### Gap N — within-tick race inflates Defeated count

When N>1 Damaged events for the same target land in one tick (very
common at 25v25 scale — 3-5 neighbours can all hit the same target
before any of them runs `set_hp`), each per-event ApplyDamage thread
reads the same `old_hp`, so the `old_hp > 0.0` transition guard fires
multiple times. Result: 745 Defeated events for 50 agents = ~15 per
agent.

Severity: COSMETIC — the simulation outcome (HP zeros, alive flips
to false, battle resolves) is correct; only the defeats_received
view inflates. Underlying issue: chronicle physics can't observe the
per-event ordering its sibling threads create. Fixing requires either
(a) a serialising aggregation pass (atomicCAS-based "first kill wins"
on alive), or (b) running ApplyDamage as a per-agent dispatch that
sums Damaged events per target THEN evaluates the transition once.
