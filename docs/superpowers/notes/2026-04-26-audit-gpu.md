# GPU Spec Audit (2026-04-26)

> Audit of `docs/spec/gpu.md` against implementation in `crates/engine_gpu/` + GPU kernel emit in `crates/dsl_compiler/`.

---

## Summary

- Layered subsystems status (per spec §1):
  - Subsystem 1 (Resident cascade): ⚠️ Partial — core pipeline implemented, several spec deviations (see §2)
  - Subsystem 2 (Sim state mirroring): ⚠️ Partial — `SimCfg` buffer done, `@cpu_only` done; tick GPU-advance done; but `PhaseTimings` missing `batch_submit_us`/`batch_poll_us`
  - Subsystem 3 (Cold-state replay): ⚠️ Partial — gold/standing/memory GPU buffers + readback done; per-event-kind dispatch-table driver **not implemented** (rules run CPU-side); `@cpu_only` annotation skips WGSL correctly
  - Subsystem 4 (Ability evaluation): ❌ Not implemented — `pick_ability` kernel absent; `per_ability` row type parsed and resolves to IR but neither CPU emitter nor WGSL emitter emits any code for it; `chosen_ability_buf` buffer absent; `apply_actions` does not read it

## Top gaps (ranked by impact)

1. **❌ `pick_ability` kernel (Subsystem 4 is entirely absent)** — The spec's §5 ability evaluation subsystem is the single largest gap. `per_ability` rows parse, resolve, and land in IR (`ScoringIR.per_ability_rows`, `PerAbilityRowIR`), but neither the CPU scoring emitter (`emit_scoring.rs`) nor the WGSL scoring emitter (`emit_scoring_wgsl.rs`) iterate over them. The `chosen_ability_buf` side buffer (`§5.7`) is not allocated. The `apply_actions` kernel does not read it. No `pick_ability_*` tests exist. `PackedAbilityRegistry.hints` and `.tag_values` are populated but bound to nothing. Impact: the entire ability-scoring GPU path is a no-op; cast decisions fall entirely to the CPU oracle / student model.

2. **⚠️ Cold-state per-event-kind dispatch-table driver (§4.1) not implemented** — The spec describes a compiler-emitted manifest + Rust driver that dispatches per-event-kind GPU handler kernels at end of each cascade. The implementation instead routes cold-state mutations (`transfer_gold`, `modify_standing`, `record_memory`) through the physics kernel's WGSL handler (atomics in `GoldBuf`, `standing_storage`, `memory_storage`). This works but bypasses §4.1's dispatch-by-event-kind design. The `cascade.rs` module header explicitly names `transfer_gold`/`modify_standing`/`record_memory` as handled by the physics kernel now (line 31–39). Tests pass because they go directly to `run_batch_resident`. The compile-time manifest check (§4.2) does not exist.

3. **⚠️ Resident fold kernels (§6.2 step 10) — CPU-only on batch path** — View fold kernels (`cs_fold_engaged_with`, etc.) are GPU-capable for the sync path but on the batch/resident path the code still calls the CPU `fold_iteration_events`. `cascade_resident.rs` module header: "Task D4 is scheduled to add a resident fold kernel — until then callers that need view storage updated must invoke the sync fold path separately." Per-tick sync fold on the batch path adds unnecessary submit+poll fences.

4. **🤔 `step_batch` returns `()` not `Result<(), BatchError>`** — Spec §2.2 declares `step_batch` returns `Result<(), BatchError>`. Implementation signature is `pub fn step_batch(...) { }` — infallible (panics via `expect()` on mid-batch kernel failures rather than propagating). `BatchError` type does not exist. Impact: callers cannot gracefully handle GPU kernel failures and fall back to sync `step()` as the spec says.

5. **⚠️ `PhaseTimings` missing spec-required `batch_submit_us` / `batch_poll_us` fields** — Spec §7.4 extends `PhaseTimings` with `batch_submit_us` (encoder cost) and `batch_poll_us` (GPU execution cost). Current struct has `gpu_cascade_us`, `gpu_seed_fold_us`, `cpu_cold_state_us`, etc., but not these two named fields. The `GpuProfiler` in `ResidentPathContext` provides per-phase timestamps via `last_batch_phase_us` (a `Vec<(&str, u64)>`) which is partially equivalent but not the typed struct fields the spec names.

## Per-section findings

### §1 Overview / layered subsystems

- **Sync mode (`step`)**: ✅ Implemented via `ComputeBackend::step`. Mask + scoring + apply_actions + movement + cascade all run GPU-side.
- **Batch mode (`step_batch`)**: ⚠️ Implemented but with deviations — returns `()` not `Result<(), BatchError>`; `pick_ability` stage absent from pipeline.
- **Non-goals**: ✅ All three non-goals (byte-exact parity in batch mode, cross-GPU reproducibility, sync path replacement) match implementation intent.

### §2 Resident cascade

- **§2.1 Principle**: ✅ Output buffers bound as next kernel's inputs; no per-tick CPU round-trips.
- **§2.2 Public surface**: ⚠️ `step_batch` exists but returns `()` not `Result<(), BatchError>`. `snapshot()` returns `Result<GpuSnapshot, SnapshotError>` as specified. `GpuSnapshot` struct matches spec exactly (`tick`, `agents`, `events_since_last`, `chronicle_since_last`).
- **§2.3 One tick on batch path**: ⚠️ Pipeline implemented through cascade. `pick_ability` step (between scoring and apply_actions) is absent. Fold kernels for views run CPU-side (not GPU-resident).
- **§2.4 Indirect dispatch**: ✅ `IndirectArgsBuffer` in `gpu_util/indirect.rs`; `run_cascade_resident_with_iter_cap` pre-records up to `iter_cap` indirect dispatches; `(0,1,1)` sentinel skips; `num_events_buf` readback post-batch.
- **§2.5 Submit shape**: ✅ One command encoder, one submit, one `device.poll(Wait)` per batch (or per chunk if `ENGINE_GPU_SUBMIT_GRANULARITY` env override set). Batch-scoped ring resets at start; `sim_cfg.tick` incremented GPU-side.
- **§2.6 Snapshot double-buffered staging**: ✅ `snapshot.rs` / `SnapshotContext` / `GpuStaging::kick_copy` + `take_snapshot` implement the spec procedure. First call returns `GpuSnapshot::empty()`.
- **§2.7 Additivity**: ✅ `ComputeBackend::step` unmodified. Parity tests run against sync path.
- **§2.8 Error cases**: ⚠️ Ring overflow error type (`SnapshotError::Ring`) exists. Staging map failure (`SnapshotError::Map`) exists. Indirect args clamp + kernel dispatch failure behaviours present. No CPU fallback on kernel dispatch failure during batch (panics instead, contrary to spec "no CPU fallback" wording but also failing the caller re-issue contract).
- **§2.9 Performance contract**: ⚠️ `chronicle_batch_perf_n100k.rs` and `chronicle_batch_perf_n200k.rs` exist as batch perf tests. The `--batch-ticks N` sweep harness with `< 0.8×` sync gate is test-harness code, not a hard CI check. `PhaseTimings` lacks the named `batch_submit_us`/`batch_poll_us` fields.
- **§2.10 Backend factoring**: ⚠️ Three-sub-struct factoring (`sync`, `resident`, `snapshot`) matches spec. `device`/`queue` are **not** `Arc<wgpu::Device>` / `Arc<wgpu::Queue>` — they are plain `wgpu::Device` / `wgpu::Queue`. `backend_label` lives in `SyncPathContext` not at top level. These are minor structural divergences.

### §3 Sim state on GPU

- **§3.1 `SimCfg` storage buffer**: ✅ `sim_cfg.rs` matches spec struct exactly (all named fields, `_reserved: [u32; 3]` — note spec shows 4 words reserved, impl has 3; struct size 64 bytes is asserted in `sim_cfg_layout.rs`).
- **§3.2 Tick advance**: ✅ Seed-indirect kernel calls `atomicAdd(&sim_cfg.tick, 1u)` at end-of-tick; no new kernel dispatch.
- **§3.3 RNG**: ✅ Pure hash; `world_seed` uploaded once at `ensure_resident_init`. Kernels read `sim_cfg.tick` instead of per-kernel cfg.
- **§3.4 CPU `state.tick` during batch**: ✅ `step_batch` does not advance `state.tick`; spec-compliant stale behavior.
- **§3.5 `@cpu_only` annotation**: ✅ Parser recognises `@cpu_only` (`parser.rs:84`). IR carries `cpu_only` flag (`ir.rs:583`). WGSL emitter skips these rules (`emit_physics_wgsl.rs:216`). Relaxed validator for banned primitives inside `@cpu_only` bodies is present (`resolve.rs:3352`).
- **§3.6 SimState mirroring summary**: ✅ All rows in the table are implemented — hot agent fields via `resident_agents_buf`, tick via `SimCfg`, world seed via `SimCfg`, cold state via side buffers, event ring via `batch_events_ring` / sync `event_ring`.
- **§3.7 Field-layout invariants**: ✅ `tests/sim_cfg_layout.rs` asserts all named field offsets. Drift panics at test time.

### §4 Cold-state replay on GPU

- **§4.1 Dispatch framework**: ❌ Spec's "one WGSL handler kernel per GPU-resident event kind" with compiler-emitted manifest + Rust glue driver is not implemented. Instead the physics kernel WGSL inline-handles gold/standing/memory mutations with atomics, which are bound at slots 17/18/19/20/21. There is no generated dispatch manifest, no per-kind event count gate, no end-of-tick secondary dispatch. `cascade.rs:31` explicitly notes the 11 remaining CPU-handled rules. Impact: the §4.1 architecture is bypassed, though the mutation semantics are approximately equivalent for the three cold-state types.
- **§4.2 Event-kind dispatch table**: ❌ No generated table. Physics kernel switch handles `EffectGoldTransfer`, `EffectStandingDelta`, `RecordMemory` inline. Compile-time manifest check does not exist.
- **§4.3 Gold**: ✅ `gold_buf` (`array<atomic<i32>>`) allocated in `ResidentPathContext`, uploaded at `ensure_resident_init`, read back on `snapshot()` (`lib.rs:2131`). Test: `cold_state_gold_transfer.rs`.
- **§4.4 Standing** (`@symmetric_pair_topk`): ⚠️ `ViewStorageSymmetricPair` GPU buffer, WGSL emitter `emit_symmetric_pair_topk_fold_wgsl`, upload/readback in place. WGSL fold kernel is compiler-emittable but the DSL `@symmetric_pair_topk` annotation parsing for `views.sim` is in `emit_view_wgsl.rs`. Hand-written `SparseStandings` deletion (spec §4.4 step 4) status unknown (out of audit scope for this session). Test: `cold_state_standing.rs`.
- **§4.5 Memory** (`@per_entity_ring`): ⚠️ `ViewStoragePerEntityRing` GPU buffer, WGSL emitter `emit_per_entity_ring_fold_wgsl`, upload/readback in place. K=64 ring with cursor. Test: `cold_state_memory.rs`.
- **§4.6 Chronicle**: ⚠️ Structured chronicle via `GpuChronicleRing` done. `@cpu_only` for narrative done. No explicit two-tier DSL grammar check — the existing chronicle-emit primitive works; `@cpu_only` is the gate.
- **§4.7 Snapshot handshake**: ✅ `gold_buf`, `standing_storage`, `memory_storage` readback into `SimState` on `snapshot()` — `lib.rs:2131+`. `chronicle_ring` watermark via `SnapshotContext`.
- **§4.8 Engine-core vs DSL-lowered**: ⚠️ Buffer alloc/binding/snapshot (engine-core) done. Rule bodies for `transfer_gold`/`modify_standing`/`record_memory` are DSL-lowered physics rules — they auto-lower to WGSL via the existing physics emitter. View fold emitters exist. However `@symmetric_pair_topk` and `@per_entity_ring` annotations are processed by `emit_view_wgsl.rs` directly, not by a "dispatch table driver reading a compiler-emitted manifest."

### §5 Ability evaluation on GPU

- **§5.1 Position in pipeline**: ❌ `pick_ability` step is absent from the batch pipeline. `step_batch` goes directly scoring → apply_actions.
- **§5.2 `pick_ability` kernel**: ❌ No `cs_pick_ability` entry point, no compiler-emitted WGSL. `physics.rs:478` notes "Unbound in Phase 1 — Phase 4 wires this."
- **§5.3 `ability::tag` / `ability::hint` / `ability::range` / `ability::on_cooldown` primitives**: ✅ All four primitives are fully parsed (`parser.rs:1266`), resolved to IR nodes (`resolve.rs:2102,2162,2440`), and emit Rust CPU code (`emit_mask.rs`, `emit_scoring.rs`). - **§5.4 `per_ability` row type**: ⚠️ Parsed (parser), resolved (resolve.rs:1119), stored in `ScoringIR.per_ability_rows`, IR type `PerAbilityRowIR` defined. **Neither CPU emitter (`emit_scoring.rs`) nor WGSL emitter (`emit_scoring_wgsl.rs`) processes `per_ability_rows`** — they silently ignore the field. No `.sim` file uses `per_ability` rows in production.
- **§5.5 Tag registry**: ✅ `PackedAbilityRegistry` has `hints: Vec<u32>` and `tag_values: Vec<f32>` (flat layout `ab * AbilityTag::COUNT + tag_idx`). `pack()` populates them from `AbilityRegistry`. Marked "Unbound in Phase 1."

- **§5.7 `ChosenAbilityBuf`**: ❌ Buffer, allocation, binding, and `apply_actions` extension to read it all absent.
- **§5.8 Engine-core vs DSL-lowered**: ❌ All items in the "engine-core" column absent. DSL-lowered `pick_ability.wgsl` absent.
- **§5.9 Failure modes**: ❌ All failure modes are moot without the kernel.

## Cross-cutting observations

1. **`per_ability` IR is a dead weight.** The parser, resolver, and IR type are complete for `per_ability` rows. The DSL user can write `row pick_ability per_ability { ... }` and it will resolve without error — but both emitters silently ignore `ScoringIR.per_ability_rows`. This is a silent no-op bug: a `.sim` file with a `per_ability` row would produce a backend that never evaluates it.

2. **Two architectural patterns coexist for cold-state GPU.** The spec §4 describes a post-cascade per-event-kind dispatch-table driver (Rust glue reading a compiler manifest). The implementation instead embeds gold/standing/memory mutations inside the physics WGSL kernel with dedicated atomic bindings. The implementation is more efficient (one kernel handles everything) but diverges from the spec's extensibility contract. This is a **spec mismatch** rather than a gap — the outcome is equivalent but the architecture differs.

3. **View fold kernels on the batch path are CPU-resident.** `cascade_resident.rs` notes "Task D4 is scheduled to add a resident fold kernel." Until then, `fold_iteration_events` runs CPU-side on the batch path, adding per-iteration sync overhead that the resident cascade design was meant to eliminate.

4. **`step_batch` signature mismatch is minor in practice but load-bearing for callers.** The spec declares `Result<(), BatchError>` so callers can catch GPU device-lost errors and fall back to sync `step()`. The current `()` return with internal `expect()` panics means a device loss during batch crashes the process rather than degrading gracefully.

5. **Kernel naming diverges from spec in spatial subsystem.** Spec §6.4.8 names three entry points `cs_spatial_hash`, `cs_kin_query`, `cs_engagement_query`. Implementation uses `cs_count`/`cs_scatter`/`cs_sort` (histogram-scatter) plus inline query in the same kernel. Functionally equivalent; no test breakage.
