# Project DAG Skill — Design Spec (v1, SUPERSEDED)

> **Status:** SUPERSEDED 2026-04-25 by `2026-04-25-project-dag-v2-design.md`.
>
> v1 designed an interactive tracking skill with a 10-invariant
> enforcement catalog. The invariant layer was overtaken by Spec B' (build
> sentinels + ast-grep CI) and Spec D + D-amendment (6 critic skills + the
> dispatch-critics orchestrator + pre-commit gate). v2 reframes Spec C as
> agent-infrastructure for a multi-day autonomous run that drains the
> roadmap, references the existing enforcement layers, and adds the agent
> loop missing from v1. Read v2 instead. v1 preserved for decision history.

## §1 Goals

1. **Two-tier task DAG** captured in TaskWarrior: roadmap items at the top, plan-task checkboxes underneath, edges via `depends:`. Replaces `docs/ROADMAP.md`'s prose tier-tracking with structured state.
2. **Architectural invariants enforced at agent-write-time.** When Claude (or any agent that goes through Claude Code's tool surface) edits a file, runs a Bash command, or finishes a session, the relevant invariants run as hooks. Violations block (severity=`block`) or warn (severity=`warn`).
3. **Compiler-first as the headline invariant.** Engine-extension tasks must produce or extend an emitter; hand-written engine handlers are blocked.
4. **Reduce doc staleness.** The skill replaces the audit cycle (we just spent ~30 minutes manually auditing 70 docs); `dag-validate` runs the same checks continuously.

## §2 Non-goals

- General-purpose Kanban for arbitrary projects (project-specific only).
- Multi-user collaboration / sync (single developer use).
- Replacing `superpowers/plans/*.md` plan files (those stay; the skill ingests their checkboxes as plan-task nodes).
- Replacing `docs/superpowers/specs/*` design specs (specs define future work; the DAG tracks execution).
- Sprint planning / time estimates / burndown (out of scope; TaskWarrior has these but the skill doesn't surface them).

## §3 Architecture

```
docs/architecture/workstreams/*.md  (workstream defs, human-edited)
            │
            ▼ informs
.claude/skills/project-dag/
  invariants.yaml                  (registry; rule + applicable paths/tags + check script)
  invariants/<rule-id>.py          (one check script per invariant)
  scripts/                         (dag-bootstrap, dag-add, dag-validate, dag-render, ...)
  hooks/                           (pre-tool-edit, post-tool-bash, session-end)
  SKILL.md                         (Claude-facing skill description)
            │
            ▼ wraps
TaskWarrior  (data: $REPO/.tasks/, git-tracked; JSON snapshot at tasks.snapshot.json)
            │
            ▼ wired into
.claude/settings.json              (PreToolUse, PostToolUse, Stop → skill hook scripts)
```

Skill is self-contained: install = drop `.claude/skills/project-dag/` + 3 hook entries in `settings.json`. Uninstall = `rm -r` + remove entries.

## §4 Data model

### §4.1 TaskWarrior schema

| Field | Type | Source | Purpose |
|---|---|---|---|
| `description` | string | std | task title |
| `status` | enum | std | pending / waiting / completed / deleted |
| `project` | string | std | workstream slug, e.g. `compiler.ability-dsl`, `gpu.megakernel` |
| `depends` | uuid list | std | DAG edges (this task waits on those tasks) |
| `tier` | string | UDA | `roadmap` or `plan-task` |
| `inv` | string list | UDA | invariant ids that apply (comma-sep), e.g. `compiler-first,schema-bump` |
| `spec_ref` | string | UDA | `<doc-path>:<section>`, e.g. `docs/spec/ability.md#§22` |
| `source_doc` | string | UDA | for plan-tasks: source plan path, e.g. `docs/superpowers/plans/foo.md` |
| `closes_commit` | string | UDA | git SHA(s) on completion (verification anchor) |

UDAs registered via `~/.taskrc` overlay loaded by the skill at init.

### §4.2 Two-tier convention

- **roadmap tier**: ~30 nodes, broad scope, sourced from `docs/ROADMAP.md`. No `depends` to other roadmap nodes (siblings, not chained); a roadmap node is "done" when all its plan-task children are.
- **plan-task tier**: ~200–500 nodes, sourced from `docs/superpowers/plans/*.md` checkboxes. Each `depends` on exactly one roadmap node (via UUID) and optionally on sibling plan-tasks.

Drift caused by direct TaskWarrior CLI use (without the skill) is allowed; `dag-validate --schema` flags malformed nodes (missing `tier`, no parent, etc.).

### §4.3 Storage

- TaskWarrior data: `$REPO/.tasks/` (set via `data.location` in repo-local `.taskrc`). Git-tracked.
- JSON snapshot: `$REPO/tasks.snapshot.json` regenerated by `dag-snapshot` and on session-end hook. Diff-friendly view of state for code review.

## §5 Workstream architecture docs

`docs/architecture/workstreams/<slug>.md` — one per top-level workstream. Initial set (~7-10):
- `compiler.ability-dsl.md`
- `compiler.economy.md`
- `gpu.megakernel.md`
- `gpu.cold-state-replay.md`
- `engine.world-sim-subsystems.md`
- `engine.debug-trace.md` (Plan 4)
- `game.ux.md`
- `tech-debt.md`

Schema (every workstream doc):

```markdown
# Workstream: <Name>
**Slug:** `compiler.ability-dsl`

## Scope
2–3 lines: what's in, what's out.

## Invariants
List of invariant ids from `invariants.yaml` that apply to all tasks in this workstream.
- `compiler-first`
- `schema-bump-on-layout`

## Entry conditions
Which task(s) gate work starting on this workstream.

## Exit conditions
Which task(s) declare the workstream done. Often a single roadmap-tier node.

## Current state
Auto-rendered by `dag-render --workstream <slug>`. Do not hand-edit.
<!-- BEGIN dag-render -->
... pipeline-injected ...
<!-- END dag-render -->
```

The "Current state" section is owned by the skill; everything else is human-edited. Skill rewrites only the content between markers on `dag-snapshot`.

## §6 Invariant catalog

### §6.1 Registry

`.claude/skills/project-dag/invariants.yaml` schema:

```yaml
- id: compiler-first
  statement: |
    Engine extension tasks must produce or extend an emitter. Hand-written
    handlers in crates/engine/src/handlers/ or generated/ are forbidden;
    behaviour comes from the DSL compiler.
  applies_to:
    paths: ["crates/engine/src/handlers/**", "crates/engine/src/cascade/handlers/**"]
    tags: ["engine-extension"]
  check: invariants/compiler-first.py
  source: docs/spec/compiler.md §1
  severity: block

- id: schema-bump-on-layout
  statement: |
    Tasks that change SoA field layout must regenerate .schema_hash.
  applies_to:
    paths: ["crates/engine/src/state/**"]
  check: invariants/schema-bump-on-layout.py
  source: docs/spec/runtime.md §22
  severity: block
```

### §6.2 Check script contract

Each `invariants/<id>.py`:
- Argv: `<event-kind>` (`edit`, `bash`, `task-add`, `task-close`, `session-end`) + event-specific args (file path, command, task UUID, etc.)
- Reads TaskWarrior state via `task export`.
- stdout: human-readable warning/notes (always shown to user).
- stderr: violation message (only shown when blocking).
- Exit codes: `0` ok, `2` block (per Claude Code hooks convention).

### §6.3 Initial catalog

Bootstrapped via Haiku scan of `docs/spec/*.md` + `docs/engine/status.md`. Expected 30-50 candidates; user filters down to load-bearing keepers (~10-15 expected for v1). Headline candidates:

1. `compiler-first` — engine handlers must come from emitter
2. `schema-bump-on-layout` — SoA layout changes require `.schema_hash` regen
3. `cross-backend-parity` — new engine behaviour needs both Serial + GPU paths (or explicit `@cpu_only`)
4. `effectop-size-budget` — `EffectOp` variants ≤ 16 bytes (per `spec/ability.md` §22)
5. `determinism-pcg` — RNG goes through `per_agent_u32(seed, agent, tick, purpose)`; no `thread_rng()` in sim code
6. `replayable-events` — events affecting state must be `is_replayable: true`
7. `mask-hash-stable` — mask predicate edits bump `MASK_HASH`
8. `scoring-hash-stable` — scoring row edits bump `SCORING_HASH`
9. `no-runtime-panic-on-deterministic-path` — sim hot path uses `Result` / saturating ops, no panics
10. `task-closes-with-commit` — completed tasks must have a non-empty `closes_commit` UDA

## §7 Skill commands

`.claude/skills/project-dag/SKILL.md` registers as `/project-dag`. Commands:

| Command | Args | Behaviour |
|---|---|---|
| `dag-bootstrap` | none (idempotent) | Scan `ROADMAP.md` → roadmap nodes; scan `superpowers/plans/*.md` → plan-task nodes + edges. Re-runnable; only creates new nodes. |
| `dag-status` | `[--workstream <slug>]` | Counts by tier × status; lists currently-blocked tasks; lists actionable leaves (no unmet deps). |
| `dag-add` | `--workstream X --tier Y --description Z [--parent uuid] [--inv id,...]` | Wraps `task add`; auto-applies invariants from workstream doc; runs `dag-validate` on the candidate before commit. |
| `dag-move` | `<id> <new-status>` | Wraps `task ... start/done/delete`. Validates exit conditions (e.g., closing a `compiler-first`-tagged task requires `closes_commit`). |
| `dag-validate` | `[--all] [--invariant id]` | Run invariants over current state. Returns per-violation report. Use `--all` for full audit; default scopes to the active task. |
| `dag-render` | `[--format dot\|mermaid] [--workstream X] [--out path]` | DAG visualization. Color by status, shape by tier, edges by depends. Default writes to `tasks.dag.svg`. |
| `dag-snapshot` | none | `task export json > tasks.snapshot.json` + re-render workstream "Current state" sections. |

Power users can call TaskWarrior directly (`task list`, `task add ...`); the skill is the recommended path because invariants run on its mutations.

## §8 Claude Code hooks

`.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      { "matcher": "Edit|Write|MultiEdit",
        "hooks": [{ "type": "command", "command": ".claude/skills/project-dag/hooks/pre-tool-edit.py" }] }
    ],
    "PostToolUse": [
      { "matcher": "Bash",
        "hooks": [{ "type": "command", "command": ".claude/skills/project-dag/hooks/post-tool-bash.py" }] }
    ],
    "Stop": [
      { "type": "command", "command": ".claude/skills/project-dag/hooks/session-end.py" }
    ]
  }
}
```

### §8.1 `pre-tool-edit.py`

Reads tool input from stdin (Claude Code passes it as JSON), extracts target `file_path`, looks up applicable invariants from `invariants.yaml`, runs each `check` script. If any returns exit 2, this hook returns exit 2 with the aggregated stderr — Claude sees the block reason and corrects.

### §8.2 `post-tool-bash.py`

Reads Bash output from stdin. Greps for `git commit` SHA pattern. If found, looks up active tasks (`+ACTIVE`) whose workstream matches the changed file paths in the commit. Prompts user (via stdout) whether to mark complete + record `closes_commit`. Always exits 0 (advisory).

### §8.3 `session-end.py`

Runs `dag-validate --all`. If new violations introduced this session, emits a session-end summary. Calls `dag-snapshot` to update `tasks.snapshot.json` and workstream `## Current state` sections. Always exits 0.

## §9 Bootstrap + maintenance

### §9.1 First-time setup

1. `mkdir -p .claude/skills/project-dag/{scripts,hooks,invariants}` and drop the skill files.
2. Add 3 hook entries to `.claude/settings.json`.
3. Create `.taskrc` with `data.location=./.tasks` and UDA registrations.
4. Run `/project-dag dag-bootstrap` — creates initial nodes from `ROADMAP.md` + `superpowers/plans/*.md`.
5. Haiku scan: spawn an audit agent that extracts invariant candidates from `docs/spec/*.md` + `docs/engine/status.md`. Review output, encode keepers into `invariants.yaml`. Write a `check` script per invariant.
6. Hand-write 7-10 workstream `.md` files (most content is the scope paragraph; invariants list comes from the encoded set).
7. `dag-snapshot` — initial commit.

### §9.2 Steady state

- New plan drafted → `dag-bootstrap` (idempotent) ingests new checkboxes.
- New invariant discovered → add to `invariants.yaml` + script. `dag-validate --all` flags any current violations.
- Task completion → `post-tool-bash.py` catches the commit SHA, prompts to close.
- Session end → `dag-validate` runs, surfaces drift before you walk away.

### §9.3 Decay protection

- The skill replaces the audit cycle. If a task is "active" but its referenced commit exists, auto-closes on next `dag-validate`. If a task is "done" but the cited commit was reverted, validate flags it.
- Workstream `## Current state` sections are skill-owned (between markers); cannot drift from TaskWarrior state.
- Hooks fire on Claude tool use, so violations get caught at write-time, not at audit-time.

## §10 Scope log

Decisions locked during brainstorming (2026-04-24):

- **D1.** Skill is project-specific (not reusable). Lives at `.claude/skills/project-dag/`.
- **D2.** Two-tier DAG: roadmap → plan-task. (Approach D from Q2.)
- **D3.** Active gatekeeper, not passive bookkeeper. Hooks block / warn at write-time. (Approach B from Q3.)
- **D4.** Audit-driven invariant catalog: Haiku scan extracts candidates, user filters. (Path ii from Q3 follow-up.)
- **D5.** TaskWarrior backing (Approach 1 from approaches list). Repo-relative `.tasks/` dir + JSON snapshot.
- **D6.** Hook scripts live inside the skill directory, not floating in `.claude/hooks/`. Skill is self-contained.
- **D7.** TaskWarrior already installed (no install step in bootstrap).
- **D8.** Hooks: `PreToolUse` on Edit|Write|MultiEdit, `PostToolUse` on Bash, `Stop` on session end.

## §11 Out of scope (explicit)

- Estimating effort / due dates (TaskWarrior supports this; we don't surface it).
- Burndown / velocity charts.
- Multi-user sync / `taskd`.
- Replacing existing `superpowers/plans/*.md` files (they remain authoritative for plan content; DAG ingests them).
- Replacing `docs/ROADMAP.md` immediately. (After v1 stabilises, ROADMAP could become a `dag-render` output instead of hand-edited prose. Out of scope for v1.)
