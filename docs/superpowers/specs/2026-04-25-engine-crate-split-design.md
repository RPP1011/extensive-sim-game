# Engine Crate Split + Sealed Rule Trait — Design Spec

> **Status:** Design (2026-04-25). Spec B from the four-layer
> architectural-enforcement framework. Makes P1 (Compiler-First) and
> related principles structurally impossible to violate at compile
> time, build time, commit time, and CI time.

## §1 Goals

1. **Restructure crates** so emitted rule logic lives in a separate crate (`engine_rules/`) from runtime primitives (`engine/`) and data shapes (`engine_data/`). Direction: `engine_data` → `engine` → `engine_rules` (downstream depends on upstream).
2. **Seal `CascadeHandler`** so impls outside `engine_rules/` don't compile.
3. **Primitives-only invariant** on `engine/`: build.rs allowlist rejects behavior modules (the Approach-2 hand-written-non-cascade-handler failure mode).
4. **Build-time + commit-time + CI enforcement** of the `// GENERATED` header rule.
5. **Stale-content detection in CI**: regenerate `engine_data/` and `engine_rules/` from DSL source; fail if diff against HEAD.
6. **Schema-hash binding**: when emitted-state layout changes, schema hash auto-bumps; CI rejects if drifted.
7. **Compile-fail tests** for sealing — assert the seal still works after every refactor.
8. **Migrate `chronicle.rs` and `engagement.rs` constants** out of engine: chronicle to emitted, engagement constants to DSL.
9. **Sweep + delete legacy hand-written rule logic** in root `src/`.

## §2 Non-goals

- Splitting `engine_gpu/` further (separate concern; out of scope).
- Restructuring DSL grammar (Spec C/E territory).
- Changing the runtime tick pipeline shape (`step_full` stays).
- Moving primitives that are *not* rule-shaped (state/, mask infra, spatial, RNG, etc., stay in engine).

## §3 Crate restructure

### §3.1 Target layout

```
crates/
  engine_data/         — was engine_generated. Renamed for clarity.
                          Data shapes only: config, entities, enums, events,
                          ids, schema, scoring, types, engagement-constants.
                          Zero engine deps.
                          Emitter target #1.
  engine/              — runtime primitives only. NO `generated/` subdir.
                          Allowlist enforced by build.rs (§5.2).
                          Depends on: engine_data.
  engine_rules/        — was empty re-export shim. Now: mask/, physics/,
                          views/, chronicle/.
                          Depends on: engine + engine_data.
                          Emitter target #2.
  engine_gpu/          (unchanged)
  dsl_compiler/        — emit destinations updated to the two crates above.
```

### §3.2 Migration moves

| From | To |
|---|---|
| `crates/engine_generated/` | `crates/engine_data/` (rename) |
| `crates/engine/src/generated/mask/` | `crates/engine_rules/src/mask/` |
| `crates/engine/src/generated/physics/` | `crates/engine_rules/src/physics/` |
| `crates/engine/src/generated/views/` | `crates/engine_rules/src/views/` |
| `crates/engine/src/chronicle.rs` (template_id → format-string switch) | `crates/engine_rules/src/chronicle/render.rs` (emitted from DSL) |
| `crates/engine/src/engagement.rs` (`break_reason` u8 constants) | `crates/engine_data/src/engagement.rs` (emitted from DSL events.sim or similar) |

`engine_rules` (the existing empty shim) gets its content rewritten; not deleted-and-recreated, to preserve a trace of git history.

### §3.3 Imports churn

Sed-able rewrite across ~30 files:
- `engine::generated::physics::*` → `engine_rules::physics::*`
- `engine::generated::mask::*` → `engine_rules::mask::*`
- `engine::generated::views::*` → `engine_rules::views::*`
- `engine_rules::*` (data types pulled via the old shim) → `engine_data::*`
- `engine::chronicle::render_entry` → `engine_rules::chronicle::render_entry`
- `engine::engagement::break_reason::*` → `engine_data::engagement::break_reason::*`

## §4 Trait sealing

### §4.1 `CascadeHandler` sealed via private supertrait

In `crates/engine/src/cascade/handler.rs`:

```rust
pub mod __sealed {
    /// Sealing trait. Only `engine_rules` may implement, because
    /// `engine_rules` blanket-impls `Sealed` for all types tagged
    /// with the `GeneratedRule` marker, which `dsl_compiler` is the
    /// only emitter of.
    ///
    /// External code that tries `impl __sealed::Sealed for Foo` is
    /// rejected by the ast-grep CI rule (§7).
    pub trait Sealed {}
}

pub trait CascadeHandler: __sealed::Sealed + Send + Sync {
    fn handle(&self, /* ... */);
}
```

In `crates/engine_rules/src/lib.rs` (auto-generated header):

```rust
// GENERATED by dsl_compiler. Do not edit.
//
// This file blanket-implements engine::cascade::handler::__sealed::Sealed
// for any type that derives the GeneratedRule marker. Combined with the
// build.rs `// GENERATED` sentinel and the ast-grep CI rule, this means
// only compiler-emitted rule types can satisfy CascadeHandler's
// supertrait bound.

#[doc(hidden)]
pub trait GeneratedRule {}

impl<T: GeneratedRule> engine::cascade::handler::__sealed::Sealed for T {}
```

Each emitted rule struct gets `impl GeneratedRule for FooHandler {}` from the emitter.

### §4.2 Other emitted-only traits

`MaterializedView`, `LazyView`, `TopKView`, mask-predicate trait — same sealing pattern, same `GeneratedRule` bound.

### §4.3 Test/demo impls

Existing demo impls in `engine/src/view/{materialized,lazy,topk}.rs` (`DamageTaken`, `NearestEnemyLazy`, `MostHostileTopK`) are PRIMITIVES used in tests. They satisfy `GeneratedRule` via `#[cfg(test)]`-gated impl in the same file:

```rust
#[cfg(test)]
impl engine_rules::GeneratedRule for DamageTaken {}
```

This admits demo impls for tests without opening the seal in production.

### §4.4 Compile-fail tests

Adds `trybuild` as a `[dev-dependencies]` entry in `crates/engine/Cargo.toml`. `crates/engine/tests/sealed_cascade_handler.rs` uses [`trybuild`](https://docs.rs/trybuild) to assert that:

```rust
// tests/ui/external_impl_rejected.rs
struct MyHandler;
impl engine::cascade::CascadeHandler for MyHandler { /* ... */ }
//~^ ERROR: the trait `engine::cascade::handler::__sealed::Sealed` is not implemented
```

fails to compile. If this test ever passes, the seal is broken.

## §5 Build-time enforcement

### §5.1 `engine_rules/build.rs` — every file must be generated

```rust
fn main() {
    let dir = std::path::Path::new("src");
    walk(dir, |path| {
        if path.file_name() == Some("lib.rs".as_ref()) { return; }
        let content = std::fs::read_to_string(path).expect("readable");
        let head: String = content.lines().take(5).collect::<Vec<_>>().join("\n");
        if !head.contains("// GENERATED by dsl_compiler") {
            panic!(
                "engine_rules/{}: missing `// GENERATED by dsl_compiler` header. \
                 Hand-edited files in this crate are forbidden.",
                path.display(),
            );
        }
    });
}
```

### §5.2 `engine/build.rs` — primitives-only allowlist

```rust
const ALLOWED_TOP_LEVEL: &[&str] = &[
    "lib.rs", "ids.rs", "creature.rs", "channel.rs",
    "rng.rs", "spatial.rs", "terrain.rs", "schema_hash.rs",
    "trajectory.rs", "step.rs", "mask.rs", "backend.rs",
];

const ALLOWED_DIRS: &[&str] = &[
    "ability", "aggregate", "cascade", "event", "invariant",
    "obs", "policy", "pool", "probe", "snapshot", "state",
    "telemetry", "view",
];

fn main() {
    let dir = std::path::Path::new("src");
    let entries = std::fs::read_dir(dir).expect("readable");
    for e in entries {
        let e = e.expect("entry");
        let name = e.file_name();
        let name_str = name.to_string_lossy();
        if e.file_type().expect("ft").is_dir() {
            if !ALLOWED_DIRS.contains(&name_str.as_ref()) {
                panic!(
                    "engine/src/{}: not in primitives allowlist. Engine contains \
                     primitives only — behavior belongs in engine_rules. To add a \
                     new primitive subdir, edit engine/build.rs ALLOWED_DIRS and \
                     justify in the AIS preamble.",
                    name_str
                );
            }
        } else if name_str.ends_with(".rs") {
            if !ALLOWED_TOP_LEVEL.contains(&name_str.as_ref()) {
                panic!("engine/src/{}: not in primitives allowlist", name_str);
            }
        }
    }
    // No `// GENERATED` markers allowed in engine/src/.
    walk_for_pattern(dir, "// GENERATED by dsl_compiler", |p| {
        panic!(
            "engine/src/{}: contains `// GENERATED` marker. Generated code lives \
             in engine_rules/, not engine/.",
            p.display()
        );
    });
}
```

The allowlist is the load-bearing structural rule: if a contributor wants to add `engine/src/theory_of_mind/`, the build fails.

**Editing the allowlist is a high-friction governance event.** It is not a routine refactor. Per user direction, adding a new entry to `ALLOWED_TOP_LEVEL` or `ALLOWED_DIRS` requires:

1. **Pros/cons writeup** in the AIS preamble of the proposing plan, justifying why the new module is a primitive (not behavior) and why no existing primitive can host it.
2. **Two independent biased-against critic agents** (Spec D infrastructure when it lands; manual via `Agent` dispatch in the meantime). Each critic is prompted with explicit bias to argue *against* the addition; both must return a `PASS` verdict for the change to advance.
3. **User approval** (recorded as an ADR per the constitution layer).

Bypassing this gate is a constitutional violation regardless of whether the build passes (the build pass means an allowlist edit was made; the question is whether the edit had the gate). Spec D will encode the critic dispatch + verdict capture; for now, this section is the contract.

### §5.3 `engine_data/build.rs` — every file generated, like engine_rules

Same pattern as §5.1.

## §6 Pre-commit hook

`.githooks/pre-commit` (~40 lines bash):

```bash
#!/usr/bin/env bash
set -e

# 1. Files staged in generated paths must have // GENERATED header.
for f in $(git diff --cached --name-only --diff-filter=AM \
           | grep -E '^crates/(engine_rules|engine_data)/src/.*\.rs$' \
           | grep -v 'lib.rs$'); do
    if ! head -5 "$f" | grep -q "// GENERATED by dsl_compiler"; then
        echo "ABORT: $f is in a generated crate but lacks the // GENERATED header." >&2
        echo "Edit the DSL source and rerun \`cargo run --bin xtask -- compile-dsl\`." >&2
        exit 1
    fi
done

# 2. Files staged outside generated paths must NOT have the marker.
for f in $(git diff --cached --name-only --diff-filter=AM \
           | grep -E '\.rs$' \
           | grep -vE '^crates/(engine_rules|engine_data)/'); do
    if grep -q "// GENERATED by dsl_compiler" "$f" 2>/dev/null; then
        echo "ABORT: $f contains // GENERATED marker but is not in a generated crate." >&2
        exit 1
    fi
done

# 3. If DSL source is staged, regen and verify generated dirs match.
if git diff --cached --name-only | grep -qE '^assets/(sim|hero_templates)/'; then
    echo "DSL source changed — regenerating to verify generated dirs are fresh..."
    cargo run --bin xtask -- compile-dsl --check >&2 || {
        echo "ABORT: generated dirs are stale relative to DSL source." >&2
        echo "Run \`cargo run --bin xtask -- compile-dsl\` and stage the changes." >&2
        exit 1
    }
fi
```

The `--check` flag is added to xtask's `compile-dsl` subcommand as part of this plan: regenerates into a temp dir, diffs against the working tree, returns non-zero if different. (Roughly 30 lines of additional Rust in xtask.)

Setup: `cargo run --bin xtask -- install-hooks` runs `git config core.hooksPath .githooks`. Documented in CLAUDE.md "Tooling caveats."

## §7 CI enforcement

### §7.1 ast-grep rules

`.ast-grep/rules/no-cascade-handler-impl-outside-engine-rules.yml`:

```yaml
id: no-cascade-handler-impl-outside-engine-rules
language: rust
rule:
  pattern: impl $$$ CascadeHandler $$$ { $$$ }
  not:
    inside:
      pattern: $A
      regex: 'crates/engine_rules/src/'
severity: error
message: |
  impl CascadeHandler must live in crates/engine_rules/. Hand-written
  cascade handlers violate P1 (Compiler-First). Edit the .sim source and
  let dsl_compiler emit the handler.
```

Similar rules for `MaterializedView`, `LazyView`, `TopKView`, mask-predicate trait, and `__sealed::Sealed`.

### §7.2 Stale-content detection (resolves Q-B from the failure-cases conversation)

CI step that REGENERATES and asserts no diff:

```yaml
- name: Regenerate DSL artefacts and verify no diff
  run: |
    cargo run --bin xtask -- compile-dsl
    git diff --exit-code crates/engine_rules/ crates/engine_data/ \
      || (echo "Generated dirs are stale — DSL source was changed but generated artefacts weren't committed." && exit 1)
```

Runs on every PR. Catches the "developer edited DSL but forgot to commit the regen" failure mode.

### §7.3 Schema-hash binding (resolves Q-F)

`crates/engine/.schema_hash` is regenerated by `compile-dsl` IF state-layout-relevant DSL changes. The CI step in §7.2 catches drift. Additionally:

```yaml
- name: Schema hash freshness
  run: cargo test -p engine --test schema_hash
```

This existing test compares `engine::schema_hash::schema_hash()` to the baseline file. If the regen step in §7.2 changed the hash but the baseline wasn't updated, this fails.

## §8 Migration scope

### §8.1 Crate renames + moves (mechanical)

1. `git mv crates/engine_generated crates/engine_data`
2. `git mv crates/engine/src/generated/* crates/engine_rules/src/`
3. Delete `crates/engine_rules/src/lib.rs` (empty shim) and rewrite based on emitter output.
4. Update `Cargo.toml` workspace members.
5. Sed cross-crate imports (~30 files; patterns in §3.3).
6. Update `dsl_compiler` emit destinations.

### §8.2 chronicle.rs migration

Move template_id catalog and `render_entry` to `engine_rules/src/chronicle/render.rs`, emitted from DSL. Templates declared in (new or extended) `assets/sim/chronicle.sim` or as part of `events.sim`. Engine API surface (`engine::chronicle::render_entry`) replaced by `engine_rules::chronicle::render_entry`.

### §8.3 engagement.rs migration

`break_reason` u8 constants move to `engine_data/src/engagement.rs`, emitted from DSL. Constants are declared alongside the events that carry them in `events.sim`. The empty `engine/src/engagement.rs` is deleted.

### §8.4 Legacy sweep (resolves Q-C from the failure-cases conversation)

User direction: delete the entire `src/` folder with the exception of the window/render code. This is decisive — `src/ai/*`, `src/world_sim/*`, `src/scenario/*`, `src/mission/*`, `src/game_core/*`, `src/content/*`, `src/narrative/*`, `src/model_backend/*`, `src/ascii_gen/*` are all legacy bootstrap superseded by the engine crate tree.

Mechanical sequence:

1. **Move `src/bin/xtask/` to `crates/xtask/`.** xtask is the project's tooling CLI (compile-dsl, scenario runner, oracle, etc.); it must survive. Audit xtask's actual imports and bring only the modules it needs into `crates/xtask/src/` (this will be far less than `src/lib.rs`'s current re-export sprawl). Update workspace `Cargo.toml`. Update root `Cargo.toml` `[[bin]]` entry to point at the new location.
2. **Move `src/rendering.rs` to a new `crates/window/` crate** (or keep at `src/rendering.rs` if root crate retains a binary). 15-line file; thin wrapper over `voxel_engine::scene`. The choice depends on whether the root `bevy_game` binary still has reason to exist.
3. **Delete everything else under `src/`.** `git rm -r src/{ai,world_sim,scenario,mission,game_core,content,narrative,model_backend,ascii_gen}` plus any leftover top-level `.rs` files that aren't in the keep-list.
4. **Trim `src/lib.rs`** to remove the re-exports of deleted modules. Likely shrinks to nothing or near-nothing — at which point the root `src/` and root `bevy_game` crate may be deletable entirely (defer that to a follow-up if it gets tangled).
5. **Verify**: `cargo build --workspace`, `cargo test --workspace`. Resolve any reference compile errors by following the trail (every reference to a deleted module is itself dead code that goes too).

Expected scope: tens of thousands of lines deleted. User flagged this as "purely a good thing." No per-file audit needed — the keep-list is `rendering` + xtask deps; everything else goes.

## §9 Sequencing

Tight ordering because moves cascade through imports:

1. Crate restructure: rename `engine_generated` → `engine_data`; create `engine_rules` content; move `generated/*` into it. Update workspace + dependencies + emitter destinations. Run `cargo test` — green.
2. chronicle.rs + engagement.rs migration. Update DSL source + regen. Run `cargo test` — green.
3. Trait sealing: add `__sealed` module + `GeneratedRule` blanket impl. Update emitted rule structs to declare `impl GeneratedRule`. Run `cargo test` — green.
4. Compile-fail test for the seal. Run `cargo test --test sealed_cascade_handler` — green (the failing case fails as expected).
5. `engine/build.rs` allowlist + `engine_rules/build.rs` + `engine_data/build.rs`. Run `cargo build` — green.
6. Pre-commit hook + xtask install command.
7. ast-grep CI rules + stale-content CI step + schema-hash freshness step.
8. Legacy sweep (Haiku audit + targeted deletes).

Each step is an independent commit; the sequencing matters because build/test green-ness is the gate.

## §10 Decision log

- **D1.** Rename `engine_generated` → `engine_data` (clarity: "data" describes content; "generated" describes process).
- **D2.** New crate is `engine_rules` (repurposing the existing empty shim).
- **D3.** Sealing pattern: private supertrait + `GeneratedRule` marker + blanket impl in `engine_rules`. Belt-and-suspenders with ast-grep + build sentinel.
- **D4.** `chronicle.rs` is rule logic, not a primitive. P1 settles it; no question.
- **D5.** `engagement.rs` constants are DSL data, not engine primitives. Emitted to `engine_data`.
- **D6.** Primitives-only allowlist on `engine/`. Adding a new primitive directory requires editing `build.rs` and a constitution check.
- **D7.** Stale-content CI: regenerate + diff. Schema-hash bumps caught by existing test.
- **D8.** Pre-commit hook ships in `.githooks/`; opt-in via `git config core.hooksPath .githooks` (one-time setup); automated via `xtask install-hooks`.
- **D9.** Legacy sweep folded into Spec B (not Spec B′). Reducing code is a goal, not a tangent. Scope: delete everything under `src/` except window code (`src/rendering.rs`) and xtask (which moves to `crates/xtask/`).
- **D11.** Allowlist edits in `engine/build.rs` require: written pros/cons, two biased-against critic-agent PASS verdicts, and user approval recorded as an ADR. The mechanism for automated critic dispatch is Spec D; until then, the gate is process.
- **D10.** `channel.rs` stays as engine primitive (dispatch table over engine enum is borderline-but-defensible; keep until evidence of growth-with-rules).

## §11 Out of scope (explicit)

- Splitting engine_gpu further.
- Renaming or restructuring dsl_compiler.
- Layer 2/3/4 of the architectural-enforcement framework (project-DAG skill, plan-template enforcement automation, fresh-context critic — separate specs).
- Refactoring root `src/` non-AI code (campaign/, mission/, viz, etc.).
- Documentation / contributor onboarding ("how to add a rule" guide).
