# Engine Crate Split + Sealed Rule Trait ‚Äî Design Spec (v2)

> **Status:** Design (2026-04-25). **Supersedes** `2026-04-25-engine-crate-split-design.md` (v1).
> Brainstorming surfaced two structural errors in v1: the dependency
> direction was inverted, and `step_full` was wrongly listed as a non-goal.
> v2 corrects both. The mid-flight Plan B1 written against v1 will be
> reshaped against v2 before further execution.

## ¬ß0 What changed from v1

1. **Dep direction inverted.** v1: `engine_data ‚Üí engine ‚Üí engine_rules` (engine_data has zero engine deps). v2: `engine ‚Üê engine_data ‚Üê engine_rules`. Engine_data depends on engine because emitted shapes are *instantiations of engine's optimized data structures* (`Memory = PerEntityRing<MemoryEntry, 64>`, `Standing = SymmetricPair<Standing, ...>`, etc.); the shape and the storage primitive are inseparable. The same semantics survive into the WGSL emit ‚Äî engine's primitives have GPU equivalents the WGSL emitter targets.

2. **`step_full` is no longer a non-goal.** v1 ¬ß2 said "Changing the runtime tick pipeline shape (`step_full` stays)." That's wrong. The phase orchestration *is* rule-aware (each phase calls a specific set of emitted operations), so `step_full` lives where the rules live: engine_rules. Engine declares the `SimBackend` trait (already present in `engine/src/backend.rs` from the GPU megakernel Phase 0); engine_rules + engine_gpu provide implementations.

3. **The primitives-vs-orchestration pattern is now uniform.** v1 had it implicitly for some pieces (cascade-handler emit) but missed it for `MaskBuffer`'s fill methods, `SimState`'s `views` field, `step_full` itself, and `register_engine_builtins`. v2 applies the pattern uniformly: engine owns *containers + raw ops*, engine_rules owns *the orchestration that wires specific rules to specific slots*.

4. **Source consolidation.** Long-term, abilities migrate from `assets/hero_templates/*.ability` into `assets/sim/*.sim`. v2 takes this as the target source layout (the `.ability` directory is transitional).

## ¬ß1 Goals

1. **Crate layering aligns with rule-awareness.** Engine = primitives. Engine_data = emitted shapes (instantiations of engine primitives). Engine_rules = emitted behavior (orchestration over engine primitives, using emitted shapes). Engine_gpu = same emitted behavior, WGSL surface. Cycle structurally impossible.
2. **`SimBackend` is the single tick-driver interface.** Engine declares the trait; backends (`SerialBackend` in engine_rules, `GpuBackend` in engine_gpu) implement it. Application code is backend-agnostic at the call site.
3. **Sealed traits enforce compiler-emit at type level.** `CascadeHandler`, `MaterializedView`, `LazyView`, `TopKView` carry a private `__sealed::Sealed` supertrait satisfiable only by the compiler-emitted `GeneratedRule` marker. Hand-written implementations don't compile.
4. **Build sentinels enforce emit at file level.** `engine_rules/build.rs` and `engine_data/build.rs` reject any file lacking `// GENERATED by dsl_compiler`. `engine/build.rs` rejects `// GENERATED` markers and constrains top-level layout to a primitives-only allowlist.
5. **Pre-commit + CI close the gap.** Pre-commit enforces the header rule on staged files and runs `compile-dsl --check` when DSL source is staged. CI regenerates and asserts no diff (catches the "edited DSL but forgot to commit regen" failure mode).
6. **Compile-fail tests assert the seals.** `trybuild` fixture proves a downstream `impl CascadeHandler` fails with `Sealed not satisfied`.
7. **Schema-hash binding.** `engine/.schema_hash` baseline regenerated on layout changes; CI fails if drifted.
8. **Migrate `chronicle.rs` and `engagement.rs` out of engine** (deferred to a follow-up plan; they contain rule-aware code).
9. **Sweep + delete legacy hand-written rule logic** in root `src/` (deferred to a follow-up plan).

## ¬ß2 Non-goals

- Splitting `engine_gpu/` further beyond what falls out of the dep-graph correction.
- Restructuring DSL grammar or extending it to carry chronicle/engagement (those migrations are planned but their grammar work is its own spec).
- Multi-user collaboration / sync / sprint planning (Spec C territory; this spec is purely the crate-layering layer).

**Note:** in v1 this section listed "step_full stays in engine" and "engine_data has zero engine deps." Both are now in ¬ß1 (corrected). They are not "out of scope" ‚Äî they are the central change.

## ¬ß3 Crate roles

### ¬ß3.1 Dependency graph

```
   engine_rules    engine_gpu       (top: emitted behavior)
        ‚Üì               ‚Üì
     engine_data ‚Üê‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò            (emitted shapes)
        ‚Üë      ‚Üò
        ‚îÇ       ‚Üò (engine uses engine_data shapes; one direction)
     engine                          (foundation: primitives)
```

`A ‚Üí B` reads "A uses B." Engine is the foundation: it defines the primitives + interfaces. engine_data uses engine's primitives to express instantiated shapes (`Memory = PerEntityRing<MemoryEntry, 64>`).

The hard rule is: **engine never imports `engine_rules` or `engine_gpu`**. The cycle v1 was concerned with ‚Äî engine ‚Üî engine_rules ‚Äî is structurally banned.

Engine's containers (`EventRing`, `CascadeRegistry`) operate on emitted types like `Event`. Under the recommended Option A (¬ß4.4), those containers are *generic* over the emitted type; engine doesn't import engine_data at all, and instantiation happens once per backend in engine_rules + engine_gpu. Under Options B or C, engine references engine_data directly via narrow allowlist exceptions or hand-written enums. Q1 (¬ß9) settles the choice during plan-writing.

### ¬ß3.3 The primitives-vs-orchestration pattern

Applied uniformly:

| Primitive (engine) | Orchestration (engine_rules, emitted) |
|---|---|
| `CascadeRegistry { table: Vec<Vec<Box<dyn CascadeHandler>>> }` + `dispatch()` | `with_engine_builtins()`: emits `register(DamageHandler); register(HealHandler); ‚Ä¶` per DSL physics rule |
| `MaskBuffer { micro_kind: Vec<bool> }` + raw `set`/`get`/`reset` | `mark_hold_allowed`, `mark_attack_allowed_from_candidates`, ‚Ä¶ ‚Äî each one wires a specific predicate to a specific bit |
| `PerEntityRing<MemoryEntry, 64>` storage | `Memory::fold_event` from emitted view |
| `EventRing` storage | event-variant emit + dispatch in handlers |
| `SimState` SoA + setters | per-handler mutators emitted as `state.set_agent_hp(...)` |
| `SimBackend` trait declaration | `SerialBackend::step` orchestration body |

Engine never knows what predicates / views / handlers exist. It knows only the *containers* and the *interfaces*. Population is engine_rules's job.

### ¬ß3.4 The `SimBackend` interface

```rust
// engine/src/backend.rs (already exists from GPU Phase 0; signature extends with `views`)
pub trait SimBackend {
    type Views;
    fn step<B: PolicyBackend>(
        &mut self,
        state:   &mut SimState,
        scratch: &mut SimScratch,
        events:  &mut EventRing<engine_data::Event>,
        views:   &mut Self::Views,
        policy:  &B,
        cascade: &CascadeRegistry<engine_data::Event>,
    );
}
```

- `engine_rules::SerialBackend`: `type Views = ViewRegistry; fn step(...) { /* phase orchestration over engine primitives + emitted rules */ }` ‚Äî body is **compiler-emitted** (¬ß3.5).
- `engine_gpu::GpuBackend`: `type Views = GpuViews; fn step(...) { /* WGSL kernel dispatch */ }` ‚Äî body is also compiler-emitted into the WGSL form.

The associated `Views` type lets each backend pick its representation: CPU uses pointer-chasing structs (`ViewRegistry { engaged_with: EngagedWith, kin_fear: KinFear, ... }`); GPU uses SoA buffers + readback. The trait is generic over `B: PolicyBackend` (existing pattern from Phase 0); the method is not object-safe (callers hold concrete backends).

### ¬ß3.5 `SerialBackend::step` is compiler-emitted (locked)

The phase ordering inside `step` is fixed (view-fold ‚Üí mask-fill ‚Üí score ‚Üí action-select ‚Üí cascade-dispatch). The *content* of each phase varies with the rule set: which views to fold, which masks to fill, which handlers register on the cascade. So the body is data-derivable from DSL declarations, and the compiler emits it.

Why emit instead of hand-write:

1. **Specialization unlocks LLVM optimization.** A hand-written generic `step` with trait-dispatch through `cascade.dispatch()` and `view_registry.fold_all()` looks like indirect calls to LLVM. An emitted `step` with literal `engine_rules::physics::damage(...)`, `engine_rules::physics::heal(...)`, `views.engaged_with.fold_event(...)`, `views.kin_fear.fold_event(...)` etc. is direct calls. LLVM inlines them, fuses adjacent loops, eliminates dead branches per known-at-emit-time literals (e.g., empty handler chains for unused event kinds become no-ops at compile time).
2. **Caching aligns with the rule set.** A given DSL source produces a deterministic `step` body; the compilation cache hits as long as the source doesn't change. Hand-written generic code re-monomorphizes on every callsite touch.
3. **Dead-rule elimination at emit time.** If a rule set doesn't use the standing view, no `fold_event` call for it appears in the emitted `step`. The cost of unused features is zero (vs. "iterate the registry, find the empty handler list, no-op" with hand-written code).
4. **GPU + CPU stay in lockstep.** The same emitter walks the rule set twice ‚Äî once for Rust (`engine_rules`), once for WGSL (`engine_gpu`). Phase ordering is identical by construction.

Plan B1' adds an `emit_step.rs` module to `dsl_compiler/` that walks the resolved IR and emits the phase body. Hand-written orchestration disappears entirely; the only `step`-shaped code in the workspace is what the compiler produces.

Application driver loop:

```rust
let mut state    = engine::SimState::new(...);
let mut scratch  = engine::SimScratch::new(...);
let mut events   = engine::EventRing::new(...);
let cascade      = engine_rules::with_engine_builtins();
let mut views    = engine_rules::ViewRegistry::new();
let policy       = ...;
let mut backend  = engine_rules::SerialBackend;  // or engine_gpu::GpuBackend behind cfg(feature = "gpu")
for _ in 0..N {
    backend.step(&mut state, &mut scratch, &mut events, &mut views, &policy, &cascade);
}
```

### ¬ß4.4 What `Event` looks like under the new direction (Option A ‚Äî locked)

`Event` enum lives in **engine_data** (emitted from DSL `event` declarations). `EventKindId` (the dense u8 ordinal ‚Äî needed by `EventRing` and `CascadeRegistry` to index by kind without knowing variants) lives in **engine**.

Engine's containers are *generic* over the event type, so engine has zero dep on engine_data:

```rust
// engine
pub struct EventRing<E: EventLike> { /* ... */ }
pub struct CascadeRegistry<E: EventLike> { /* ... */ }
pub trait EventLike { fn kind(&self) -> EventKindId; }

// engine_data (GENERATED)
pub enum Event { AgentMoved { ... }, AgentAttacked { ... }, ... }
impl engine::EventLike for Event { fn kind(&self) -> EventKindId { /* match arms */ } }

// engine_rules (uses both)
pub type SimEventRing = engine::EventRing<engine_data::Event>;
```

Concrete instantiation happens in engine_rules + engine_gpu. Application driver code that previously constructed `engine::EventRing::new(...)` constructs `engine_rules::SimEventRing::new(...)` (or a re-export). The genericity cost is bounded ‚Äî one type parameter per container ‚Äî and the optimizer monomorphizes everything at the instantiation point, so the runtime cost is zero.

Considered and rejected:
- **Option B** (host `Event` in engine via a `// GENERATED` allowlist exception): would break the "no `// GENERATED` in engine" invariant for one file. Pragmatic but corrosive ‚Äî once one exception lands, others get justified by analogy.
- **Option C** (hand-written `Event` in engine validated by the compiler): loses the "events are emitted" property; new variants would require hand-edits + ADRs, defeating the rules-as-data goal.

Option A: `EventRing<E>` and `CascadeRegistry<E>` both generic; engine declares the bounds, engine_rules instantiates `EventRing<engine_data::Event>`. Engine has no concrete Event dep.

Option B: engine has a generated submodule. The `engine/build.rs` allowlist gets one exception: `engine/src/generated_event.rs` is allowed and required to carry `// GENERATED`. Engine's runtime primitives reference `crate::generated_event::Event` directly. Pragmatic but breaks the "no GENERATED in engine" invariant.

Option C: hand-write a fixed `Event` enum in engine that the DSL must conform to. The compiler validates that every DSL `event` declaration matches an existing variant; new variants require a hand-edit + an ADR. Most rigid.

**My read: A** (generic primitives). It's the most aligned with "engine has no rules"; the genericity cost is one type parameter per container; instantiation happens once per backend in engine_rules / engine_gpu. Plan B1' draws this out.

## ¬ß6 Trait sealing

(Carries from v1 ¬ß4.)

### ¬ß6.1 `CascadeHandler` sealed via private supertrait

```rust
// engine/src/cascade/handler.rs
pub mod __sealed { pub trait Sealed {} }
pub trait CascadeHandler: __sealed::Sealed + Send + Sync {
    fn handle(&self, /* ... */);
}
```

### ¬ß6.2 `GeneratedRule` marker + blanket impl

```rust
// engine_rules/src/lib.rs (allowlisted; not GENERATED)
pub trait GeneratedRule {}
impl<T: GeneratedRule> engine::cascade::handler::__sealed::Sealed for T {}
```

Each emitted rule struct gets `impl crate::GeneratedRule for FooHandler {}` from the emitter.

### ¬ß6.3 Other emitted-only traits

`MaterializedView`, `LazyView`, `TopKView` follow the same pattern. The blanket impl in `engine_rules/src/lib.rs` extends to satisfy the sealing supertrait of all four traits.

### ¬ß6.4 Test/demo impls

Existing demo impls in `engine/src/view/{materialized,lazy,topk}.rs` (`DamageTaken`, `NearestEnemyLazy`, `MostHostileTopK`) are PRIMITIVES used in tests. They satisfy `GeneratedRule` via `#[cfg(test)]`-gated impl in the same file:

```rust
#[cfg(test)]
impl engine_rules::GeneratedRule for DamageTaken {}
```

This admits demo impls for tests without opening the seal in production. (Test files in `crates/engine/tests/` follow the same pattern for hand-written test handlers.)

### ¬ß6.5 Compile-fail tests

`crates/engine/tests/sealed_cascade_handler.rs` uses `trybuild` to assert that an external `impl CascadeHandler` fails with `Sealed not satisfied`.
