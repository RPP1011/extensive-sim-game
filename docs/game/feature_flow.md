# Feature flow

How to add a new game feature. The path is compiler-first: every feature enters the codebase as compiler-emitted Rust from DSL source. There is no hand-write-then-compile loop; there is no bootstrap crate. If the compiler can't emit the feature yet, extend the compiler.

## Two scenarios

**(A) The compiler already supports the declaration kinds your feature needs.** Most features will land this way once the compiler matures. Write DSL source, compile, test.

**(B) The compiler does not yet support one of the declaration kinds you need.** Extend the compiler first (grammar + lowering + emission), then write the DSL source. This is where the compiler milestones from `compiler_progress.md` get advanced.

Both paths end the same way: a commit that lands DSL source + compiler-emitted Rust (checked in for readability) + a fixture test asserting behavior.

## Scenario A — feature in supported kinds

```
1. Write the DSL source
   └─ assets/sim/<feature>.sim

2. Compile
   └─ cargo run --bin xtask -- compile-dsl
      emits crates/engine_rules/src/<feature>.rs (and Python mirror)

3. Write a fixture test
   └─ crates/engine_rules/tests/<feature>.rs
      seeded scenario → step N ticks → assert exact event log

4. cargo test -p engine_rules
5. Commit (DSL + emitted Rust + test as one unit)
```

The emitted Rust is committed. It's readable, reviewable, and the same file the compiler will regenerate on the next `compile-dsl` invocation. Reviewers diff both the DSL source and the emitted output; if the emitted output looks wrong, the fix is in the compiler's emitter, not in the emitted file.

## Scenario B — compiler extension required

```
0. Identify the missing declaration kind (or the missing shape within an
   existing kind — e.g. `@materialized` views if only `@lazy` is supported).

1. Extend the compiler
   └─ crates/dsl_compiler/src/
      - parser:   recognize the new syntax
      - lower:    AST → IR
      - emit_rust / emit_python: IR → target code
   
2. Add a golden test for the new emission shape
   └─ crates/dsl_compiler/tests/<kind>_emit.rs
      tiny hand-written .sim input → expected .rs output (string-equal diff)

3. Confirm milestone row in docs/game/compiler_progress.md flips to 🚧

4. Proceed through scenario A steps above for the actual feature.

5. On commit: milestone row flips to ✅ IF this feature's fixture tests
   pass AND the legacy code it replaces is deleted in the same commit.
```

Extending the compiler is slower than writing Rust directly. That is the cost of the compiler-first discipline. The return: every line that lands is permanent, and the emission shape is proven by being used.

## Legacy replacement

Many features "already exist" as hand-written Rust in `crates/engine/src/ability/*.rs`, `creature.rs`, `policy/utility.rs`, and balance constants in `step.rs`. These are legacy and do not grow. When a compiler milestone covers a legacy handler, the SAME commit that lands the compiler support must delete the legacy handler. No parallel paths.

Fixture tests that currently exercise the legacy code will now exercise the compiler-emitted code. They must still pass bit-for-bit.

If a feature spans multiple declaration kinds (an event + a physics rule + a mask + a scoring entry — e.g. the engagement update spans all four), all of them move together. Partial migration (event+physics moved, mask+scoring still legacy) is not a valid state — the mask predicate won't compile against an event type that no longer has an emitter.

## Where emitted files land

| Declaration kind | Emission target | Rationale |
|---|---|---|
| `event` | `crates/engine_rules/src/events/` | Pure data; no engine deps. Engine re-exports the enum. |
| `physics` | `crates/engine/src/generated/physics/` | Handlers reference `engine::cascade::CascadeHandler`, `engine::state::SimState`, `engine::event::EventRing`. Putting them in `engine_rules` would invert the dep direction (engine already depends on engine_rules); putting them in `engine` keeps the cycle clean. The aggregator `register(registry)` is called from `CascadeRegistry::register_engine_builtins`. |
| `mask` | `crates/engine/src/generated/mask/` | Same dep argument as physics. |
| `scoring` | `crates/engine_data/src/scoring/` | Split out of `engine_rules` (commit `4137f02b`, 2026-04-22) into a dedicated generated-only crate so engine compile times stay fast. |

## Shape contract for emitted code

Emitted physics files (`crates/engine/src/generated/physics/`) follow a fixed shape so reviewers can skim them:

```rust
// GENERATED by dsl_compiler from assets/sim/<feature>.sim.
// Edit the .sim source; rerun `xtask compile-dsl`. Do not edit by hand.

use engine::cascade::{CascadeHandler, CascadeRegistry, EventKindId, Lane};
use engine::event::{Event, EventRing};
use engine::state::SimState;

pub struct <Feature>Handler;

impl CascadeHandler for <Feature>Handler {
    fn trigger(&self) -> EventKindId { EventKindId::<Feature> }
    fn lane(&self) -> Lane { Lane::Effect }

    fn handle(&self, event: &Event, state: &mut SimState, events: &mut EventRing) {
        // body is the mechanical lowering of the `physics` rule
    }
}

pub fn register(registry: &mut CascadeRegistry) {
    registry.register(Box::new(<Feature>Handler));
}
```

Reviewers: if the emitted code has helper functions, macros, or trait abstractions that the DSL didn't ask for, push back. The compiler's output should be verbose and mechanical — that's what makes it auditable.

## Forbidden shortcuts

- **Do not hand-author `crates/engine_rules/src/*.rs`.** Those files are generated. If you need a rule the compiler can't emit, extend the compiler.
- **Do not add game logic to `crates/engine/`.** The engine contains primitives (cascade runtime, SoA, spatial, event ring, RNG, schema hashing). Adding `ATTACK_DAMAGE = 12.0` to `step.rs` is a regression even if "temporary."
- **Do not skip the fixture test.** Every feature lands with a seeded scenario + expected event log. Without it, the legacy → DSL-owned transition has no safety net.
- **Do not mix compiler-extension and feature-add in one commit.** Land the compiler extension first (with a tiny golden test), then use it for the actual feature. Keeps review reviewable.

## Schema-hash implications

Adding an event, entity field, scoring row, or physics rule bumps the relevant schema sub-hash (see `compiler.md` §2). Non-append changes (rename, reorder, type change) block merge unless a new trace-format baseline lands at the same time.

Schema hashes are emitted by the compiler from the DSL source. Manually tampering with the emitted hash values is a review-blocker.
