`Plan 3.5` (engagement/ZoC, 2028 LoC, 8 tasks, commit `30b06bf6`), and
`Ability Plan 1` (combat + world effects, 2329 LoC, 13 tasks, commit
`d8253d6a`, pulled from a parallel worktree). Combined: ~5400 LoC of plan
material, most of it code-sample detail.

**What's not yet written.** Plans 4 (debug & trace), 5 (ComputeBackend trait),
6 (GpuBackend foundation), 7+ (per-kernel GPU porting) per `status.md:33-36`.
All are prerequisites for the cross-backend parity contract in
`docs/engine/spec.md:52-57`.

**Highest-leverage consolidation.** Plan 3.5 and Ability Plan 1 both add
SoA fields to `SimState` and register new cascade handlers that interact
with combat range. Plan 3.5 §Task 4 explicitly shares `ENGAGEMENT_RANGE=2.0`
with `ATTACK_RANGE`, and the plan's own follow-up §Follow-ups #1 acknowledges
"Ability Plan 1 integration... the engagement-aware cast gate lands as the
first task of an Ability-Plan-1 integration PR". Executing them back-to-back
on one branch amortizes one schema-hash bump, one `status.md` update, and one
`step_full` integration.

**Spec drift found.** Engine's SoA needs fields are `hunger` / `thirst` /
`rest_timer` (`crates/engine/src/state/mod.rs:22-24`). `docs/dsl/state.md:94-101`
commits to the 6-dim Maslow-inspired set:
`hunger / safety / shelter / social / purpose / esteem`. Only `hunger`
matches. `thirst` and `rest_timer` are **not** in state.md's catalog;
`safety`, `shelter`, `social`, `purpose`, `esteem` are in state.md but
**not** in the engine. Secondary drift: state.md requires `armor`, `magic_resist`,
`attack_damage`, `attack_range`, `shield_hp`, `status_effects`, `level`
as Combat/Vitality fields (state.md:39-48); the engine has only `hp` + `max_hp`;
Ability Plan 1 adds `shield` and timed status timers but still skips `armor` /
`attack_damage` / `attack_range` / `magic_resist`.

**Recommended next action.** Resolve the needs-dim spec drift first (is
state.md authoritative, or does the engine reflect a simplified MVP?). Then
merge Plan 3.5 into Ability Plan 1 as a new Task 0 (engagement state
landing before cast state, so the cast gate already reads it). Then execute
the merged plan on one branch, followed by Plan 3 last because it must
serialize **all** SoA fields added by the combined plan. See §6 Proposal 1.

## 4. Draft-plan overlap analysis

Pairwise overlap between the three drafts.

### Plan 3 (persistence) vs Plan 3.5 (ZoC)

- **SoA field overlap:** Plan 3's snapshot writer hard-codes the current
  SoA field list at plan line ~272-280 (`hot_pos`, `hot_hp`, `hot_max_hp`,
  `hot_hunger`, `hot_thirst`, `hot_rest_timer`, `hot_movement_mode`). Plan 3.5
  adds `hot_engaged_with` (plan line 21). **If Plan 3 is executed before
  Plan 3.5, the snapshot format does not cover the engagement field** —
  saved games lose engagement state on reload. Fix would be a schema-hash
  bump on Plan 3.5's side and migration.
- **Schema-hash overlap:** both plans bump `.schema_hash`. Plan 3 Task 11
  bumps for snapshot format + obs packer + probe. Plan 3.5 Task 8 bumps for
  `hot_engaged_with`, new event variant, new EventKindId. Two sequential
  merges produce two baseline-bump commits; landing them as one plan allows
  one atomic bump.
- **Cascade overlap:** none. Plan 3 registers no cascade handlers; Plan 3.5
  adds one (`OpportunityAttackHandler`).
- **Finding:** Plan 3 must land **after** Plan 3.5, or the snapshot writer
  needs a post-facto amendment. Currently `status.md:29` lists Plan 3 ahead
  in the Plans index, but that's historical order — not dependency order.

### Plan 3.5 (ZoC) vs Ability Plan 1 (combat)

- **SoA field overlap:** disjoint.
  - Plan 3.5: `hot_engaged_with: Vec<Option<AgentId>>`.
  - Ability Plan 1: `hot_shield`, `hot_shield_expires_at`, `hot_stun_remaining`,
    `hot_slow_remaining`, `hot_slow_factor_q8`, `hot_cooldown_ready_at`,
    `hot_gold`, `cold_standing`.
- **Constant overlap:** `ENGAGEMENT_RANGE = 2.0` (Plan 3.5 Task 3) is the
  **same value as** `ATTACK_RANGE = 2.0` already in `step.rs`. Plan 3.5 plan
  text line 11 explicitly shares this constant: "this plan deliberately
  shares the `ENGAGEMENT_RANGE = 2.0` constant with `ATTACK_RANGE`".
  Ability Plan 1 does NOT touch `ATTACK_RANGE`. No collision, but coupling:
  any future change to the attack melee distance must also touch engagement.
- **Pipeline phase overlap:** both plans touch the **top** of the tick loop.
  - Plan 3.5: inserts `update_engagements(state)` at the start of phase 1
    (plan line 23).
  - Ability Plan 1 Task 9: inserts `ability::expire::tick_start_decrement`
    "**before** policy evaluation" (plan line 76, §Acceptance item 3
    re-confirms). This is also phase 1 or earlier.
  - **Ordering decision needed.** If stun expires first, a just-unstunned
    agent can engage (correct). If engagement updates first, a slowed agent's
    engagement state is current (correct). Both orderings seem benign but
    determinism requires the order be frozen — the merged plan must pick one
    and add a regression test.
- **Cascade handler overlap:** disjoint handlers, but shared event budget
  (single `EventRing` with fixed capacity). Ability Plan 1 alone emits up to
  13 new event kinds per tick per agent (worst case: a chained `CastAbility`
  at depth 8 = 9 AgentCast + 8 effect events). Plan 3.5 adds 1 more
  (`OpportunityAttackTriggered`). Combined worst-case budget tightens.
- **Mask overlap:** disjoint. Ability Plan 1 adds `cast_valid` mask head;
  Plan 3.5 doesn't extend the mask.
- **Acceptance-fixture overlap:** Ability Plan 1 Task 12 acceptance "2v2 cast"
  could trivially engage if Plan 3.5 lands first — meaning all 2v2 casts
  from melee range fire off engagement + opportunity attacks.
  Plan 3.5 Task 7 "tank wall protects wizard" doesn't involve abilities.
  The two acceptance suites don't conflict but **Ability Plan 1's 2v2 test
  outcome changes** if engagement is in play. Running Ability Plan 1's
  acceptance before 3.5 lands, then after, would produce different event
  hashes — so one of the plan's determinism promises (same-seed-same-hash)
  would not survive a "re-run after 3.5 landed" check unless the fixture is
  written non-engageable.
- **Follow-up explicitly acknowledged:** Plan 3.5 §Follow-ups #1
  (plan line 1982):
  > "Ability Plan 1 integration. When `CastHandler` lands ..., its
  > melee-range cast gate needs to read `state.agent_engaged_with(caster)`
  > and emit `OpportunityAttackTriggered` for any non-engaged-target cast
  > fired while engaged. Expected change: ≤10 lines in `CastHandler::handle`,
  > plus one `engagement_gated_cast_emits_opportunity_attack` test. Add to
  > Ability Plan 1 as the first task of its integration PR."

### Ability Plan 1 vs state.md

- **Does Ability Plan 1 cover state.md's Combat/Vitality?** **Partially.**
  state.md Combat/Vitality (state.md:39-48) lists 8 fields (`hp`, `max_hp`,
  `shield_hp`, `armor`, `magic_resist`, `attack_damage`, `attack_range`,
  `status_effects`). Engine has 2 (`hp`, `max_hp`); Ability Plan 1 adds 1
  (`shield`) + 4 status-effect timer slots (`stun_remaining`, `slow_remaining`,
  `slow_factor_q8`, `shield_expires_at`) as a flat replacement for
  `Vec<StatusEffect>`. Missing: `armor`, `magic_resist`, `attack_damage`,
  `attack_range`. Ability Plan 1 §Out of scope line 2306-2314 explicitly
  defers `Vec<StatusEffect>` stack + richer effects to Plan 2.
- **Does the plan's IR match state.md?** Ability Plan 1 hand-authors
  `AbilityProgram = Vec<EffectOp>` in the engine (plan line 7: "tactical_sim::effects
  types are not imported"). This is a parallel shape to the existing

## 6. Consolidation recommendations

Five concrete proposals. Each includes a YES / NO / CONDITIONAL verdict.

---

> **Proposal 1: Merge Plan 3.5 into Ability Plan 1.**
>
> **Rationale:** Both drafts touch combat-adjacent state + cascade handlers.
> Plan 3.5 §Follow-ups #1 already declares the integration point (≤10 lines in
> `CastHandler::handle` plus one test). Ordering is deterministic-sensitive
> (tick-start decrement vs `update_engagements`), and a merged plan lets the

> also stay coherent — the 2v2 cast test knows engagement exists.
>
> **Impact:** Plan count drops from 3 drafts → 2 drafts. Task count
> reduces from 8 + 13 = 21 → ~19 (Plan 3.5's T2 `is_hostile_to` stub
> merges cleanly into Ability Plan 1's existing creature setup; schema-hash
> bumps merge into Ability Plan 1's T13). One `status.md` row covers both
> systems. One `cargo test -p engine` run validates the combined surface.
>
> **Risk:** merged plan grows from 2329 to ~4200 LoC. Becomes a single
> long-running execution effort; partial-completion points are harder to
> identify. Mitigation: preamble explicitly splits the merged plan into
> "engagement first (T0-T3)" then "ability (T4-T15)" with a commit at
> each boundary.
>
> **Recommendation: YES** — conditional on the author drafting a short
> 50-line integration preamble at the top of the merged plan that (a)
> pins the tick-pipeline phase order (stun-expiry first, then
> `update_engagements`, then policy eval) with a regression test and (b)
> writes the engagement-aware cast gate as its own task rather than as

>

>
> **Risk:** delays having save/load. If the user needs save/load for

> conditional recommendation.
>
> **Recommendation: CONDITIONAL** — YES if ability work lands in the next
> 2-3 sessions; NO if persistence is blocking scenario iteration. Ask
> the user.

---

> **Proposal 3: Update `docs/dsl/state.md` OR the engine to resolve the
> needs-6-dim drift — do not keep both.**
>

> (3 dims, non-Maslow). state.md:89-101 commits to Maslow-6

> engine; any code review that cites the engine will flag state.md.

> - **Option A:** update state.md's Needs section to match the engine's
>   three-dim set (`hunger / thirst / rest`). Implies state.md is the

> - **Option B:** add five new hot SoA fields to the engine
>   (`safety`, `shelter`, `social`, `purpose`, `esteem`) and either drop
>   `thirst`/`rest_timer` or rename them. Expands the engine MVP scope.

>

>
> **Recommendation: YES, pick one.** Author's lean: Option B deferred;

> the goal-evaluation / aspiration work state.md requires.
> Decide via open question (§9.1).
