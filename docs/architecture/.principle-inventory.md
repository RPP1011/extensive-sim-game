## Extracted candidates (20 total)

### Compilation & emission

- ★ **Unified engine, dual emission** — Compiler has one target (the engine); all output (scalar Rust, SPIR-V, Python) lands in the engine or external ML consumers. No CPU vs GPU target split at compiler level. | source: `docs/spec/compiler.md` §1

- ★ **Scalar Rust is reference** — `SerialBackend` is the reference implementation and determinism oracle for all tests, parity checks, and verified-correct ports. | source: `docs/spec/runtime.md` §1

- **Dual backends, same interface** — `SerialBackend` (host-resident scalar Rust) and `GpuBackend` (Vulkan SPIR-V kernels) both implement the same public API. Test code is backend-agnostic; internally profoundly different. | source: `docs/spec/runtime.md` §3, §1

- ★ **Cross-backend parity is correctness invariant** — Implementation is correct iff it satisfies spec AND produces byte-identical `replayable_sha256()` across both backends on the same seed. Parity is first-class correctness, not aspiration. | source: `docs/spec/runtime.md` Intro

- **GPU kernels ship with Serial closures** — Every DSL mask predicate, cascade handler, and view gets both scalar Rust (registered with `SerialBackend`) AND SPIR-V kernel (registered with `GpuBackend`). | source: `docs/spec/compiler.md` §1.2, §1

- **Emission modes are backend-local** — Scalar Rust, GPU dispatch code, SPIR-V kernels, and Python dataclasses are all produced from one DSL source; the engine selects backend at init. | source: `docs/spec/compiler.md` §1

- **Python emission is trace-consumer, not runtime** — Python dataclasses + pytorch `Dataset` mirror entities/events/`@traced` views; all training-script concerns (algorithm, reward shaping, observation packing) live in external code. | source: `docs/spec/compiler.md` §1

### Determinism & verification

- ★ **Determinism contract is total** — Same seed + same `schema_hash` + same agent spawn order + same action sequence ⟹ bit-exact SHA-256 over replayable-subset event log, on every platform, across both backends. | source: `docs/spec/runtime.md` §2

- **RNG constants pinned across backends** — All shader randomness derives from `WorldRng` (PCG-XSH-RR) with four fixed-keyed constants embedded in Rust AND in shader source via specialization constants or preprocessor defines. Golden tests assert per-agent-u32 returns identical values on host and in compute shader. | source: `docs/spec/runtime.md` §7

- **Event ordering is deterministic within ticks** — GPU atomic-append race resolved by sorting events by stable per-tick sequence number before hashing — both backends emit same events in same logical order. | source: `docs/spec/runtime.md` §2, §5

- **HashMap forbidden in hot paths** — `HashMap` iteration is forbidden in hot paths on both backends; use `BTreeMap` (serial) or sorted indices (GPU). | source: `docs/spec/runtime.md` §2

- **Float reductions require fixed-point or sort** — Float reductions use either integer fixed-point OR sorted-key accumulation to avoid associativity drift. Applies to both backends; GPU must sort events by `target_id` before atomic reduction. | source: `docs/spec/runtime.md` §2, §1.2

- **Action shuffle is host-side and deterministic** — Phase 3 shuffle uses seed-deterministic Fisher-Yates on ActionBuffer after download (GPU) or in-place (Serial). Download-shuffle-upload cost is negligible relative to one shuffle per tick. | source: `docs/spec/runtime.md` §14

### Schema & versioning

- ★ **Schema hash gates save/load** — Combined hash covers state + event + rules + scoring + RNG constants + kernel catalog hashes. Checkpoint load rejects on mismatch; kernel recompile bumps hash automatically. | source: `docs/spec/runtime.md` §22, `docs/spec/compiler.md` §2

- **Schema changes must append** — Non-append changes (remove, reorder, type change) block merge without explicit schema bump. CI guard enforces. | source: `docs/spec/compiler.md` §2

- **Trace schema mismatch is hard error** — Loading a trace whose `combined_hash` differs from current DSL is a hard error with detailed diff (field changes, event changes, git remediation hints). | source: `docs/spec/compiler.md` §2

### Layering & scope

- ★ **Emergence over primitives** — A new entity / event / `EffectOp` requires demonstrating it cannot arise from existing primitives. Most macro behavior is pure emergence. | source: `docs/spec/economy.md` §1.3

- **Recipes are abilities** — Economic actions compose from the ability DSL's `EffectOp` vocabulary. No parallel action system. | source: `docs/spec/economy.md` §1.3

- **ML is out of DSL scope** — Policy architecture, training algorithms, curriculum, reward shaping, observation packing live in external pytorch code. Compiler emits Python dataclasses + pytorch `Dataset`; all training concerns are out-of-scope. | source: `docs/spec/language.md` §10, `docs/spec/compiler.md` §1

- **Cascade rules are compiled, not runtime config** — Compiler emits SPIR-V kernels + Rust closures for cascade handlers; engine provides the dispatch runtime. Rules themselves are DSL-authored, not engine-configured. | source: `docs/spec/runtime.md` §1, §11

### Weak-test risks

- **Announce distance semantics unspecified** — All announce tests place observers on same z-plane as speaker, so 3D vs planar is indistinguishable. Spec is silent; implementation uses 3D. | source: `docs/engine/status.md` Open verification questions #1
