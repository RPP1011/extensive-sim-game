Missions & Scenarios

Multi-room dungeons bridge the turn-based campaign overworld and the deterministic 100ms-tick combat simulation. This page covers mission structure, procedural room generation, scenario configuration, and the runner that executes it all.

1. Mission Structure

The game's architecture follows a three-layer simulation: 

Campaign (turn-based overworld) → Mission (multi-room dungeons) → Combat (100ms fixed-tick deterministic sim)

A mission is a sequence of rooms that the player's party progresses through. Each room triggers one or more combat encounters via the core step(state, intents, dt_ms) function in the simulation engine. Missions are the connective tissue between the strategic campaign layer (see Campaign) and the tactical combat layer (see Combat Loop).

Each mission is tracked by a MissionSnapshot struct that captures all runtime state: name, bound region, turns remaining, reactor integrity, sabotage progress, alert level, current room index, and tactical mode.

Mission Components

In the ECS layer, a mission entity is decomposed into three components:

ComponentPurpose
MissionDataStatic identity: ID, name, bound region
MissionProgressMutable runtime state: active flag, result, turns remaining, sabotage/alert/integrity gauges, room index
MissionTacticsTactical directives: mode (Balanced/Aggressive/Defensive), command cooldown, forced orders

An optional AssignedHero component tracks which hero companion is assigned to a mission entity, and the ActiveMission marker identifies the mission currently receiving focused player attention.

2. Room Types & Map Layout

Missions are organized around a MissionMap resource containing an ordered list of RoomMetadata entries. Each room has a type that governs its narrative role, threat budget, and sabotage threshold (the progress required to enter that room):

Room TypeDimensionsRole
Entry20 × 20Opening encounter; low threat, introduces the dungeon
Pressure16 × 16Tight space, escalating danger, enemy reinforcements
Pivot18 × 18Decision point; branching paths, environmental interactions
Setpiece32 × 32Large-scale encounter with environmental mechanics
Recovery18 × 18Respite room; stress reduction, healing opportunities
Climax30 × 30Boss encounter; highest threat budget, mission finale
Open100 × 100Open-field arena; used by scenarios and drills

Each room also carries interaction nodes — environmental verbs (e.g., "Drain", "Flood", "Dispel", "Collapse") that the player can trigger for tactical advantage at some cost. For example, the default "Emberwell Reliquary" mission map includes rooms like Moon Gate Vault (Entry, threat 8), Floodgate Walk (Pressure, threat 14), and Ritual Nave (Climax, threat 32).

3. Room Generation

Rooms are procedurally generated by the room_gen module (src/mission/room_gen/). The system produces a RoomLayout struct containing:

  • NavGrid — row-major walkability grid with elevation data, converted to a GridNav for pathfinding
  • SpawnZones — player and enemy spawn positions, quality-scored by nearby cover and elevation
  • Obstacles — walls, pillars, barricades, L-shapes, cover clusters, sandbags, platforms, and ramps

Generation Pipeline

The generator uses a deterministic LCG (linear congruential generator) seeded from the scenario seed. It selects 2–3 primitive obstacle generators and layers them via composite obstacle generation:

  1. Room dimensions are looked up from the RoomType
  2. A NavGrid is created with all cells initially walkable
  3. Composite obstacles are placed: pillar grids, wall segments with gaps, L-shape cover, barricade lines, cover clusters, sandbag arcs, and platform edges
  4. Spawn zones are positioned at opposite ends of the room
  5. The layout is validated (connectivity, spawn reachability)

The roomgen CLI

You can export generated room layouts to NDJSON using the xtask CLI: 

cargo run --bin xtask -- roomgen export --output generated/rooms.jsonl

This exports room data including nav grids, obstacle placements, and spawn positions for visualization, debugging, or external tooling. The ml_gen submodule provides hooks for ML-based room generation experiments.

4. Scenario Configuration

Combat encounters are defined in TOML scenario files parsed into a ScenarioFile struct containing a ScenarioCfg and an optional ScenarioAssert block.

ScenarioCfg Fields

FieldTypeDefaultDescription
nameString—Scenario identifier
seedu64—RNG seed for deterministic replay
hero_countusize4Number of hero units (ignored when hero_templates is set)
enemy_countusize4Number of generic enemy units
difficultyu322Difficulty tier; scales enemy stats via scale_enemy_stats
max_ticksu643000Timeout in simulation ticks (each tick = 100ms, so 3000 = 5 min)
room_typeString"Entry"Room type for procedural layout generation
hero_templatesVec<String>[]Named hero templates from assets/hero_templates/
enemy_hero_templatesVec<String>[]Hero templates for the enemy team (hero-vs-hero mode)
hp_multiplierf321.0Multiplies all unit HP to adjust time-to-kill
manifest_pathOption<String>NonePath to ability manifest TOML for resolving ability_refs

Drill-Specific Fields

Scenarios can also function as training drills for the AI with additional fields:

  • drill_type — identifies the drill (e.g., "reach_point", "kite_enemy", "kill_healer")
  • target_position — [x, y] goal for movement drills
  • enemy_units — custom enemy definitions with template, behavior, tag, position, and stat overrides
  • hazards — static hazard zones (damage, slow, heal) with position, radius, team, timing
  • objective — drill objective ("reach_position", "survive", "kill_all", "kill_target", "protect_ally")
  • action_mask — restricts available action heads ("move_only", "move_attack", "all")

Assertion Block

Scenarios can include an [assert] section for automated testing:

  • outcome — expected result: "Victory", "Defeat", or "Timeout"
  • max_ticks_to_win — must win within this many ticks
  • min_heroes_alive / max_heroes_dead — survival constraints

Example Scenario File

[scenario]
name = "cross_mixed_5v5"
seed = 7005
hero_count = 5
enemy_count = 0
difficulty = 1
max_ticks = 3000
room_type = "Entry"
hero_templates = ["brawler", "striker", "fighter", "shadow", "archmage"]
enemy_hero_templates = ["sniper", "stunner", "medic", "paladin", "templar"]

Hero-vs-hero mode: when enemy_hero_templates is set, both teams use full hero templates with abilities and passives instead of generic scaled enemies. Set enemy_count = 0 to avoid spawning additional generic enemies.

5. Objectives & Win/Loss Conditions

Each mission tracks multiple gauges that determine its outcome:

GaugeDescription
sabotage_progressObjective completion meter; reaching sabotage_goal (default 100) completes all objectives
reactor_integrityWard lattice health (0–100); dropping to zero triggers defeat
alert_levelEnemy awareness; higher alert increases enemy damage and reinforcement rate
turns_remainingTime limit; reaching zero triggers defeat

Victory Conditions

  • All MissionObjective components marked as completed (triggered when sabotage progress reaches the goal)

Defeat Conditions

  • Any hero's health reaches zero
  • Reactor integrity drops to zero (ward lattice collapsed)
  • Turns remaining reaches zero (time expired)

The MissionResult enum tracks the outcome: InProgress, Victory, or Defeat.

6. Room Transitions

Room progression within a mission is driven by the mission_map_progression_system. As the player's sabotage_progress crosses a room's sabotage_threshold, the party advances to the next room in the MissionMap sequence: 

// Simplified from mission_systems.rs
if progress.sabotage_progress >= next_room.sabotage_threshold {
    progress.room_index = next_room_index;
    // "Entered 'Reliquary Fork' (Pivot)"
}

Each room type shapes the encounter differently. Recovery rooms reduce stress, Pressure rooms are tight corridors with high alert, and Climax rooms feature the highest threat budgets with environmental interaction nodes for boss encounters.

7. Scenario Runner

The scenario runner (src/scenario/runner.rs and src/scenario/combat_setup.rs) converts a ScenarioCfg into a live combat encounter. Two entry points are available:

  • run_scenario_to_state(cfg) — builds a SimState + SquadAiState with open-field spawns (no room layout)
  • run_scenario_to_state_with_room(cfg) — same but with a procedurally generated RoomLayout and GridNav for terrain-aware pathfinding

Build Pipeline

  1. Resolve templates — hero template names are looked up from assets/hero_templates/, dataset/hero_templates/, assets/lol_heroes/, or dataset/heroes/ subdirectories
  2. Load manifest — optional ability manifest resolves ability_refs and passive_refs into full ability definitions
  3. Build SimState — units are spawned at calculated positions; three modes: heroes vs generic enemies, hero-vs-hero, or drill mode with custom enemy definitions
  4. Scale enemies — difficulty-based stat scaling via scale_enemy_stats
  5. Apply HP multiplier — adjusts time-to-kill across all units
  6. Initialize AI — SquadAiState is inferred from unit stats

CLI Execution

# Run a single scenario
cargo run --bin xtask -- scenario run scenarios/basic_4v4.toml

# Benchmark all scenarios in a directory
cargo run --bin xtask -- scenario bench scenarios/

# Generate scenario dataset
cargo run --bin xtask -- scenario generate dataset/scenarios/

# Oracle evaluation
cargo run --bin xtask -- scenario oracle eval scenarios/

Scenario Results

Each run produces a ScenarioResult with: outcome ("Victory"/"Defeat"/"Timeout"), tick count, assertion pass/fail status, final unit counts, event log, hero death count, and per-unit combat statistics including damage dealt/taken, healing, shields, CC stats, and per-ability breakdowns.
