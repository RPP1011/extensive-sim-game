The Combat Loop

The deterministic step() function drives all combat. Every 100ms tick advances the simulation through a fixed sequence of phases, producing a stream of events that fully describe what happened.

The step() Function

The entire combat simulation is driven by a single pure function:

pub fn step(mut state: SimState, intents: &[UnitIntent], dt_ms: u32) -> (SimState, Vec<SimEvent>)

It takes the current SimState, a slice of UnitIntent (what each unit wants to do this tick), and a time delta (always FIXED_TICK_MS = 100). It returns the new state and a list of events that occurred.

Pure function. The same inputs always produce the same outputs. No global state, no side effects, no external RNG. This is the foundation of the determinism contract.

Tick Phases

Each call to step() executes these phases in strict order:

state.tick += 1;

// Phase 1: System ticks (order matters)
tick_hero_cooldowns(&mut state, dt_ms);        // Decrement ability cooldowns
tick_status_effects(&mut state, tick, dt_ms);   // Tick DoTs, HoTs, expire buffs/debuffs
advance_projectiles(&mut state, tick, dt_ms);   // Move projectiles, check collisions
tick_periodic_passives(&mut state, tick, dt_ms); // Fire periodic passive triggers
tick_zones(&mut state, tick, dt_ms);             // Tick persistent ground zones
tick_channels(&mut state, tick, dt_ms);          // Advance channel casts
tick_tethers(&mut state, tick, dt_ms);           // Check tether ranges, apply tether ticks

// Phase 2: Per-unit processing (shuffled order)
for unit in shuffled_units {
    // Process casting, auto-attacks, movement, intent execution
}

Phase 1: tick_hero_cooldowns

Decrements cooldown_remaining_ms on all ability and passive slots for every living unit. Respects cooldown reduction buffs (the cooldown_reduction stat buff multiplier).

Phase 2: tick_status_effects

Iterates all active status effects on every unit. Ticks DoT/HoT effects at their tick_interval_ms, decrements remaining duration, and removes expired effects. Fires StatusEffectExpired events.

Phase 3: advance_projectiles

Moves all in-flight projectiles by speed * dt_ms / 1000. Checks for collision with targets. On hit, applies on_hit effects and fires ProjectileHit events. Piercing projectiles continue past targets.

Phase 4: tick_periodic_passives

Checks all passive abilities with Periodic triggers and fires them if their interval has elapsed.

Phase 5: tick_zones

Ticks persistent ground zones. Applies effects to units inside the zone at each tick_interval_ms. Removes expired zones.

Phase 6: tick_channels

Advances channeled abilities. Applies channel tick effects and checks for interrupts. Channels end early if the caster is stunned or killed.

Phase 7: tick_tethers

Checks tether connections. If target moves beyond max_range, the tether breaks. Otherwise applies tick effects and fires on_complete effects when the tether expires naturally.

Phase 8: Per-Unit Processing

After system ticks, each living unit is processed. The processing order is shuffled per tick using the deterministic RNG to prevent first-mover advantage. Each unit:

  1. Checks if it's crowd-controlled (stunned, etc.) — if so, skips action
  2. Continues any in-progress cast (decrements remaining_ms, completes if zero)
  3. Processes its UnitIntent (attack, use ability, move, heal, etc.)
  4. Auto-attack cooldown decrements
  5. Resource regeneration (resource_regen_per_sec * dt_ms / 1000)

SimState

The full simulation state, passed into and returned from step():

pub struct SimState {
    pub tick: u64,                    // Current tick number
    pub units: UnitStore,             // All units (alive and dead)
    pub projectiles: Vec<Projectile>, // In-flight projectiles
    pub rng_state: u64,               // Deterministic RNG seed
    pub zones: Vec<Zone>,             // Persistent ground effects
    pub obstacles: Vec<Obstacle>,     // Terrain obstacles
    pub tethers: Vec<Tether>,         // Active tethers
    pub duels: Vec<Duel>,             // Active duels
    // ...
}

The Determinism Contract

All simulation randomness flows through SimState.rng_state via next_rand_u32(). This is a XorShift64 generator:

pub fn next_rand_u32(state: &mut SimState) -> u32 {
    let mut x = state.rng_state;
    x ^= x >> 12;
    x ^= x << 25;
    x ^= x >> 27;
    state.rng_state = x;
    (x.wrapping_mul(0x2545_f491_4f6c_dd1d) >> 32) as u32
}

Never use thread_rng() or any external RNG source in simulation code. This would break determinism. All randomness must come from SimState.rng_state.

Determinism guarantees:

  • Same initial state + same intents sequence = identical outcome
  • Unit processing order is shuffled per tick, but the shuffle itself is deterministic (seeded from rng_state)
  • Tests in src/ai/core/tests/determinism.rs verify reproducibility
  • CI runs determinism tests in both debug and release modes

The Fixed Tick

pub const FIXED_TICK_MS: u32 = 100;

The simulation runs at a fixed 10 ticks per second (100ms per tick). This is not frame-rate dependent — the renderer interpolates between ticks for smooth visuals. The fixed tick ensures determinism regardless of hardware speed.
