Summons & Directed Units

How the Summon effect creates new units on the battlefield, including clones that mirror the caster and directed summons that attack on command.

1. The Summon Effect

The Summon effect variant creates one or more new units during combat. It is defined with the following parameters:

ParameterTypeDefaultDescription
templateStringrequired Name of the summon template to instantiate (looked up via build_summon_template). Use "self" to clone the caster.
countu321 Number of units to summon.
hp_percentf32100.0 Percentage of max HP the summoned unit spawns with. A value of 50.0 means the summon appears at half health.
cloneboolfalse If true, the summon is a clone of the caster rather than a template-based unit.
clone_damage_percentf3275.0 When clone is true, the clone's attack_damage is scaled to this percentage of the caster's damage.
directedboolfalse If true, the summon does not act independently. It attacks only when its owner attacks, from its own position.

DSL Example

effect summon {
  template = "skeleton_warrior"
  count = 3
  hp_percent = 80
}

effect summon {
  template = "self"
  clone = true
  clone_damage_percent = 50
}

2. Summoned Units

Every summoned unit exists as a full UnitState in the simulation. It shares the same tick loop, ability system, and status effect pipeline as any other unit. What distinguishes a summon from a regular unit are two fields on UnitState:

  • owner_id: Option<u32> — set to Some(caster_id) for all summons. This links the summon to its creator.
  • directed: bool — when true, the summon follows special attack rules instead of acting on its own AI.

At spawn time, the engine assigns a fresh unique id to each summoned unit (computed as max_existing_id + 1). The summon is placed on the caster's team and positioned near the caster with an incremental X offset of 1.0 + (i * 0.5) for each additional summon beyond the first.

Lifetime. Summoned units persist until killed. They have no built-in duration timer — if you want temporary summons, pair the Summon effect with a timed zone or a delayed self-damage effect on the summon template.

3. Directed Summons

When the directed flag is set to true, the summoned unit behaves fundamentally differently from a normal unit:

  • No independent AI. Directed summons do not choose their own targets or move on their own. Their move_speed_per_sec is set to 0.0 at spawn.
  • Attack-on-owner-attack. When the owner unit performs an attack, all of its directed summons also attack the same target from their own positions. This means range matters — a directed summon that is too far away will not reach the target.
  • Ground-targeted placement. If the ability uses position targeting (AbilityTarget::Position), directed summons spawn at the targeted position rather than next to the caster.

Design pattern: Azir-style soldiers. Directed summons are the engine's implementation of "pet soldiers" that stand where you place them and stab enemies when you command an attack. Use CommandSummons to reposition them afterward.

4. Clones

When clone is true (or when the template is "self"), the engine creates the summon by cloning the caster's entire UnitState rather than looking up a template. The clone receives:

  • All of the caster's stats (HP, armor, magic resist, attack range, etc.)
  • All of the caster's abilities
  • A fresh id and cleared status effects, shields, and state history
  • HP scaled to hp_percent of the caster's max_hp
  • Attack damage scaled to clone_damage_percent of the caster's attack_damage (default 75%)

Clones are full units that act independently (unless also marked directed). They can cast abilities, be targeted by enemies, and receive status effects just like any other unit.

5. CommandSummons

The CommandSummons effect repositions all directed summons owned by the caster. It moves every owned directed summon (alive, with owner_id == caster_id and directed == true) to a target position instantly.

ParameterTypeDefaultDescription
speedf328.0 Movement speed parameter. In the current implementation, repositioning is instant (soldiers dash to the location).

The target position is resolved from AbilityTarget:

  • AbilityTarget::Position(p) — summons move to position p.
  • AbilityTarget::Unit(id) — summons move to that unit's current position.
  • AbilityTarget::None — summons move to the caster's position.

DSL Example

effect command_summons {
  speed = 10
}

6. UnitSummoned Event

Every time a summon is created, the simulation logs a UnitSummoned event: 

SimEvent::UnitSummoned {
    tick: u64,        // simulation tick when the summon appeared
    unit_id: u32,     // the new unit's unique ID
    template: String, // template name (or "self" for clones)
}

This event can be used by the AI decision pipeline to track summon counts, by the replay system to visualize spawns, and by scenario scoring to evaluate summon-heavy strategies.
