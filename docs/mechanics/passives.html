Passives & Triggers

Data-driven passive abilities that fire automatically when game events occur. Passives are defined as plain data and resolved by the simulation engine at hook points.

PassiveDef

A PassiveDef is the static definition of a passive ability. It is pure data — no closures or callbacks. The simulation reads these definitions at runtime and evaluates them against incoming events.

FieldTypeDefaultDescription
nameString—Human-readable name. Logged in PassiveTriggered events.
triggerTrigger—When this passive fires. See the comprehensive trigger table below.
cooldown_msu320Internal cooldown. After firing, the passive cannot trigger again for this many milliseconds.
effectsVec<ConditionalEffect>[]List of conditional effects to apply when triggered. Each can have its own condition, area, tags, and stacking mode.
rangef320.0Used by Periodic triggers to find the lowest-HP ally within range as the default target.
// Example PassiveDef: heal lowest-HP ally every 3 seconds
PassiveDef {
    name: "Healing Aura",
    trigger: Trigger::Periodic { interval_ms: 3000 },
    cooldown_ms: 0,
    effects: vec![ConditionalEffect { effect: Heal { amount: 15, .. }, .. }],
    range: 6.0,
}

PassiveSlot

PassiveSlot is the runtime wrapper around a PassiveDef, stored on each UnitState in the passives vector. It tracks the live cooldown and periodic timer state.

FieldTypeDescription
defPassiveDefThe static passive definition.
cooldown_remaining_msu32Remaining cooldown in milliseconds. The passive can only fire when this is 0.
periodic_elapsed_msu32For Periodic triggers only: accumulates dt_ms each tick. When it reaches interval_ms, the passive fires and the counter resets.

The is_ready() method returns true when cooldown_remaining_ms == 0.

All Trigger Types

The Trigger enum defines every event that can cause a passive to fire. Triggers are split into two categories: event-driven triggers (handled by check_passive_triggers) and periodic triggers (handled by tick_periodic_passives).

Trigger Resolution

Passive triggers are resolved by two separate systems depending on their type:

Event-Driven: check_passive_triggers

Called after specific game events (damage, kills, heals, status changes, etc.). The function:

  1. Depth guard: Checks passive_trigger_depth. If it is 3 or greater, the function returns immediately to prevent infinite recursion (a passive triggering another passive triggering another, etc.).
  2. Increments depth: passive_trigger_depth += 1.
  3. Iterates all passives on the triggering unit, checking:
    • Is the passive off cooldown? (cooldown_remaining_ms == 0)
    • Does the passive's trigger type match the incoming event type?
    • For range-gated triggers (OnAllyDamaged, OnAllyKilled): is the context target within range?
    • For threshold triggers (OnHpBelow, OnHpAbove): does the current value meet the threshold?
    • For OnStackReached: does the stack name match and is the count at or above the threshold?
  4. On match: Emits a PassiveTriggered event, sets the passive's cooldown, and applies each ConditionalEffect (evaluating conditions via evaluate_condition_tracked).
  5. Decrements depth: passive_trigger_depth -= 1.
// Trigger matching is exhaustive pattern-match:
(Trigger::OnDamageDealt, Trigger::OnDamageDealt) => true,
(Trigger::OnHpBelow { percent: threshold },
 Trigger::OnHpBelow { percent: current_pct }) => current_pct <= threshold,
(Trigger::OnAllyDamaged { range },
 Trigger::OnAllyDamaged { .. }) => distance(unit_pos, ally_pos) <= range,
_ => false,

Recursion limit: The depth cap of 3 means a passive can trigger another passive, which can trigger a third, but no further. This prevents infinite chains like "on damage dealt, deal damage" loops.

Periodic: tick_periodic_passives

Called once per simulation tick in the main step() loop, after tick_status_effects. For each living unit, it iterates passives looking for Trigger::Periodic { interval_ms }:

  1. Skip if passive is on cooldown.
  2. Accumulate: periodic_elapsed_ms += dt_ms.
  3. If periodic_elapsed_ms >= interval_ms:
    • Emit PassiveTriggered event.
    • Find the lowest-HP ally within the passive's range as the default target (falls back to self).
    • Apply each ConditionalEffect, resolving area targets if an area is specified.
    • Reset: periodic_elapsed_ms -= interval_ms (preserves sub-interval remainder).
    • Set cooldown: cooldown_remaining_ms = cooldown_ms.
  4. Otherwise, just store the updated periodic_elapsed_ms.

Passive Cooldowns

Every PassiveDef has a cooldown_ms field (default 0). When a passive fires, PassiveSlot.cooldown_remaining_ms is set to this value. The cooldown is decremented each tick in tick_hero_cooldowns. While cooldown_remaining_ms > 0, the passive is skipped during trigger checks.

A cooldown of 0 means the passive can fire every tick (or every time its event occurs). This is intentional for effects like "on every auto-attack, apply a stack" but should be used carefully to avoid performance issues with high-frequency triggers.

Design tip: Always set a meaningful cooldown on passives with broad triggers like OnDamageTaken or OnDamageDealt. Without a cooldown, a DoT effect ticking every 100ms would fire the passive 10 times per second.

PassiveTriggered Event

Every time a passive fires, the simulation emits a PassiveTriggered event for logging, replay, and AI observation: 

SimEvent::PassiveTriggered {
    tick: u64,          // Simulation tick when the passive fired
    unit_id: u32,       // ID of the unit whose passive triggered
    passive_name: String, // Name from the PassiveDef
}

This event is emitted before the passive's effects are applied, so downstream event handlers can see the trigger event followed by the resulting damage/heal/status events in sequence.
