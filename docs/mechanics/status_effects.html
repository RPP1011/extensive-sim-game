Status Effects

Every buff, debuff, DoT, HoT, shield, crowd-control effect, and status interaction tracked on a unit at runtime. Status effects are the primary mechanism for modifying unit behaviour and stats during combat.

ActiveStatusEffect

Each live status effect on a unit is stored as an ActiveStatusEffect struct in the unit's status_effects vector. The simulation's tick_status_effects function decrements remaining_ms every tick and removes expired entries.

FieldTypeDescription
kindStatusKindDiscriminated union describing the effect type and its parameters.
source_idu32ID of the unit that applied the effect. Used for kill-credit on DoT deaths.
remaining_msu32Milliseconds remaining. Decremented by dt_ms each tick; effect is removed when it reaches 0.
tagsTagsNamed power-level tags. Compared against the target's resistance_tags to determine if the effect is resisted.
stackingStackingHow duplicate applications interact: Refresh (reset timer), Extend (add duration), Strongest (keep higher value), or Stack (allow multiple instances).

StatusKind — All Variants

StatusKind is the core enum that determines what a status effect actually does. The simulation pattern-matches on it during tick_status_effects to apply per-tick logic (DoT damage, HoT healing, shield expiry, etc.) and during intent processing to enforce crowd control.

Buffs

A Buff is created by the Effect::Buff variant, which specifies a stat name, a multiplicative factor, and a duration_ms. When applied, it inserts a StatusKind::Buff { stat, factor } into the target's status_effects vector. 

// Effect definition
Effect::Buff { stat: "attack_damage", factor: 1.3, duration_ms: 5000 }

// Creates this status on the unit:
StatusKind::Buff { stat: "attack_damage", factor: 1.3 }
// with remaining_ms = 5000

During combat resolution, code that reads a unit's stats checks for active Buff status effects and multiplies the base stat by each matching factor. Multiple buffs to the same stat stack multiplicatively.

Debuffs

Debuffs work identically to buffs but reduce stats. The Effect::Debuff variant produces a StatusKind::Debuff { stat, factor }. A factor of 0.7 on move_speed reduces movement speed to 70% of base. 

// Reduce target's armor by 40% for 4 seconds
Effect::Debuff { stat: "armor", factor: 0.6, duration_ms: 4000 }

Damage over Time (DoT)

DoT effects deal periodic damage via the StatusKind::Dot variant. Three internal fields control the ticking:

  • amount_per_tick — damage applied each time the timer fires.
  • tick_interval_ms — how many milliseconds between ticks (e.g. 500 ms = 2 ticks/sec).
  • tick_elapsed_ms — internal accumulator. Each sim tick adds dt_ms; when it reaches tick_interval_ms, damage is applied and the counter resets.

The tick_status_effects function handles this. When a DoT tick kills the unit, a UnitDied event is emitted and OnDeath/OnKill passive triggers fire (credited to source_id). 

// In tick_status_effects:
StatusKind::Dot { amount_per_tick, tick_interval_ms, tick_elapsed_ms } => {
    *tick_elapsed_ms += dt_ms;
    if *tick_elapsed_ms >= *tick_interval_ms {
        *tick_elapsed_ms -= *tick_interval_ms;
        // Apply amount_per_tick damage to unit
    }
}

Kill credit: DoT deaths credit the source_id stored on the ActiveStatusEffect, so the original caster receives the kill even if they are no longer alive.

Healing over Time (HoT)

HoTs mirror DoTs exactly, using StatusKind::Hot with the same three-field ticking mechanism. The only difference is that they add HP (capped at max_hp) instead of subtracting it. When healing fires, tick_status_effects emits a HealApplied event and triggers OnHealReceived passives on the target.

Stacks

The stack system lets abilities build up named counters on a target, which other abilities can then consume or react to.

ApplyStacks Effect

The Effect::ApplyStacks variant adds stacks to a unit:

FieldTypeDefaultDescription
nameString—Unique identifier for this stack type (e.g. "poison", "mark").
countu321Number of stacks to add per application.
max_stacksu324Hard cap. Further applications are clamped.
duration_msu320Duration for the stack status effect.

At runtime, stacks are stored as StatusKind::Stacks { name, count, max_stacks } on the target's status_effects list.

OnStackReached Trigger

The Trigger::OnStackReached { name, count } passive trigger fires when a unit's named stack count reaches or exceeds the specified threshold. This enables "mark-and-detonate" patterns. See the Passives page for full trigger documentation.

Dispel

The Effect::Dispel removes status effects from a target. It accepts an optional target_tags filter:

  • If target_tags is empty, all removable status effects are cleared.
  • If tags are specified, only effects whose tags contain a matching key are removed.
// Remove all debuffs with the "magic" tag
Effect::Dispel { target_tags: vec!["magic".into()] }

// Remove all status effects
Effect::Dispel { target_tags: vec![] }

Immunity

The Effect::Immunity variant grants temporary immunity to effects matching specific tags. It creates a StatusKind::Immunity { immune_to } status on the target.

FieldTypeDescription
immune_toVec<String>List of tag names. Incoming effects with any matching tag are blocked.
duration_msu32How long the immunity lasts.

When an effect is applied to a unit with an active Immunity status, the sim checks whether any of the effect's tags appear in the immune_to list. If so, the effect is discarded and an EffectResisted event is emitted.

Status Transfer & Clone

StatusTransfer

Effect::StatusTransfer moves status effects between the caster and target.

FieldTypeDefaultDescription
steal_buffsboolfalseIf true, steals buffs from the target and applies them to the caster. If false, transfers debuffs from the caster onto the target.

StatusClone

Effect::StatusClone copies status effects from the target onto the caster (or vice versa), up to max_count effects (default: 3). Unlike transfer, the original effects remain on the source.

Detonate

Effect::Detonate consumes all Stacks status effects on the target and deals burst damage based on the consumed stack count.

FieldTypeDefaultDescription
damage_multiplierf321.0Damage per consumed stack. Total burst = stack_count * damage_multiplier.

This enables the classic "apply stacks, then detonate" gameplay pattern. Combine with ApplyStacks on basic attacks and Detonate on an active ability for a mark-and-pop combo.

Resistance Tags

Every unit has a resistance_tags: Tags field (a HashMap<String, f32>). When an effect is applied, its tags are compared against the target's resistance_tags using the check_tags_resisted function: 

pub fn check_tags_resisted(effect_tags: &Tags, resistance_tags: &Tags) -> bool {
    for (tag, &power) in effect_tags {
        if let Some(&resistance) = resistance_tags.get(tag) {
            if resistance >= power {
                return true;  // Effect is resisted
            }
        }
    }
    false
}

If any single resistance tag meets or exceeds the effect's corresponding power level, the entire effect is blocked and an EffectResisted event is emitted: 

SimEvent::EffectResisted {
    tick,
    unit_id: target_id,
    resisted_tag: "immune:stun",   // tag that caused the resist
}

Example: An effect with {"CROWD_CONTROL": 80.0} applied to a unit with resistance_tags: {"CROWD_CONTROL": 90.0} is resisted (90 ≥ 80). If the resistance were only 50.0, the effect would go through.
