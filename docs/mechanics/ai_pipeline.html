AI Decision Pipeline

A layered override system where each AI layer can override the one above it — from team-wide squad strategy down to individual neural urgency interrupts. The final intent for each unit is what gets fed to step().

Overview

Intent generation flows through five layers, each of which can override the decision from the layer above:

Layer 1: Squad AI        (team personality & formation)
   ↓ override
Layer 2: Ability Evaluator (neural urgency interrupt, fires when urgency > 0.4)
   ↓ override
Layer 3: Transformer      (cross-attention decision head)
   ↓ override
Layer 4: GOAP             (goal-oriented action planning)
   ↓ override
Layer 5: Control AI       (hard CC timing coordination)
   ↓
Final UnitIntent → step()

Each layer is optional. A minimal setup might only use Squad AI. The full pipeline uses all five for maximum tactical intelligence.

The Intent System

Every unit's desired action for a tick is represented as a UnitIntent. Intents are collected from the AI pipeline and passed to step() as a slice:

pub fn step(state: SimState, intents: &[UnitIntent], dt_ms: u32) -> (SimState, Vec<SimEvent>)

Common intent types include:

  • Attack a specific target (auto-attack)
  • UseAbility on a target or position (hero ability by index)
  • Move to a position
  • Heal an ally
  • Idle (do nothing this tick)

Layer 1: Squad AI

The highest-level AI operates at the team/squad level. It assigns broad behavior based on personality profiles and formation modes.

Personality Profiles

Each squad has a personality that biases all unit decisions:

PersonalityBehavior
AggressivePrioritize damage, dive backline, focus kills
DefensiveProtect allies, peel for carries, hold formation
BalancedAdapt to situation, trade efficiently
SkirmishHit-and-run, kite, disengage when pressured

Formation Modes

Formation determines spatial behavior — how the squad positions itself on the grid:

  • Line: Tanks front, ranged behind
  • Spread: Units spread to avoid AoE
  • Cluster: Group for shared buffs/heals
  • Flank: Send mobile units around the sides

Source: src/ai/squad/intents.rs

Layer 2: Neural Ability Evaluator

An urgency-based interrupt layer that can override Squad AI's decision when an ability use is critical.

Urgency threshold: The evaluator fires when urgency > 0.4. Below that threshold, Squad AI's intent stands. Above it, the evaluator's ability recommendation takes priority.

The evaluator is a small neural network that scores each available ability based on:

  • Current combat state (ally/enemy positions, HP, cooldowns)
  • Ability properties (damage, CC duration, range)
  • Tactical context (is an ally about to die? is an enemy grouped for AoE?)

It produces an urgency score (0.0–1.0) and a recommended ability + target. Only abilities with urgency above the threshold override the squad-level intent.

Source: src/ai/core/ability_eval/

Layer 3: Transformer Decision Head

A grokking-based transformer that processes entity tokens (all units on the field) and ability tokens (available abilities) through cross-attention to make decisions.

Architecture

  • Entity tokens: Each unit is encoded as a feature vector (HP%, position, team, status effects, etc.)
  • Ability tokens: Each available ability is encoded (cooldown state, damage, range, effect type)
  • Cross-attention: Ability tokens attend to entity tokens to determine optimal target selection
  • Output: Action distribution over (ability, target) pairs

The transformer uses a pointer action space — it points to specific entities on the field rather than choosing from a fixed action set. This allows it to generalize across different team compositions.

Source: src/ai/core/ability_transformer/

Layer 4: GOAP Planner

Goal-Oriented Action Planning provides structured decision-making through goals and actions.

How GOAP Works

  1. Goals define desired world states (e.g., "enemy_healer_dead", "ally_HP_above_50%")
  2. Actions define how to change world state (e.g., "attack_healer", "cast_heal")
  3. The planner searches for the cheapest sequence of actions to achieve the highest-priority goal
  4. The first action in the plan becomes the unit's intent

Party Culture

GOAP is modified by party culture — a set of modifiers that bias goal priorities and action costs based on the squad's overall composition and personality. A party with many healers might deprioritize "self-heal" goals in favor of "protect-ally" goals.

GOAP also has its own DSL (.goap files) for defining goals and actions declaratively.

Source: src/ai/goap/

Layer 5: Control AI

The lowest-level override, specialized for hard CC timing coordination. When a crowd control opportunity arises, the Control AI can override all other layers to ensure CC is applied at the optimal moment.

Key behaviors:

  • CC chaining: Stagger stuns so they don't overlap, maximizing total lockdown time
  • Interrupt priority: Use CC to interrupt enemy channels at the right moment
  • Peel timing: Save CC for when an ally is in immediate danger

Self-Play Reinforcement Learning

The AI layers (especially Layer 2 and 3) are trained via self-play RL:

AlgorithmUse
REINFORCEPolicy gradient baseline for simple ability decisions
PPOProximal Policy Optimization for stable transformer training

The self-play loop:

  1. Two AI squads play against each other using the current policy
  2. Combat outcomes generate reward signals (damage dealt, kills, survival, healing)
  3. Gradients update the policy network
  4. The improved policy plays against itself, creating a curriculum of increasing difficulty
