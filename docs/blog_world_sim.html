The Goal

We wanted a zero-player sandbox — a world that generates its own stories. NPCs with inner lives form grudges, fall in love, start businesses, betray allies, and die in sieges they didn't see coming. No scripted events. No quest designers. Just systems interacting with systems until history happens.

The constraint: it has to run fast. Fast enough to simulate thousands of years of history during world generation, and fast enough to keep ticking in real-time once the player shows up.

This post is a technical walkthrough of how the world sim works, from the tick architecture down to the NPC decision loop.

Architecture: Snapshot → Compute → Merge → Apply

Every tick follows a four-phase pipeline:

┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ Snapshot │ ──▶ │ Compute │ ──▶ │  Merge  │ ──▶ │  Apply  │
│  freeze  │     │   167   │     │ commut. │     │ produce │
│  state   │     │ systems │     │  fold   │     │  next   │
│          │     │         │     │         │     │  state  │
└─────────┘     └─────────┘     └─────────┘     └─────────┘

Why not just mutate in place? Two reasons: parallelism and determinism.

If system A writes to an NPC's HP while system B reads it, you get data races or order-dependent results. Instead, every system reads the same frozen snapshot and emits pure data — WorldDelta enums. No closures, no references to mutable state. Just plain structs describing what changed.

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum WorldDelta {
    Damage { target_id: u32, amount: f32, source_id: u32 },
    Heal { target_id: u32, amount: f32, source_id: u32 },
    Move { entity_id: u32, force: (f32, f32) },
    ProduceCommodity { settlement_id: u32, commodity: usize, amount: f32 },
    UpdateEntityField { entity_id: u32, field: EntityField, value: f32 },
    AddBehaviorTags { entity_id: u32, tags: [(u32, f32); 8], count: u8 },
    RecordChronicle { entry: ChronicleEntry },
    SpawnEntity { kind: EntityKind, pos: (f32, f32), team: WorldTeam, level: u32 },
    Die { entity_id: u32 },
    // ... 30+ more variants
}

The merge phase folds all deltas commutatively — damage sums, movement forces add as vectors, deaths union into a set, position teleports are last-write-wins. Because the fold is commutative, the order systems run in doesn't affect the result. The simulation is deterministic regardless of thread scheduling.

The Merge: Zero-HashMap Flat Arrays

The naive approach — HashMap<EntityId, AccumulatedChanges> — allocates on every insert. With 2,000+ entities and 167 systems producing deltas, that's thousands of hash lookups per tick.

Instead, we pre-allocate flat arrays indexed by entity ID:

struct FlatMergedDeltas {
    // Per-entity combat (indexed by entity ID)
    damage: Vec<f32>,
    heals: Vec<f32>,
    shields: Vec<f32>,
    force_x: Vec<f32>,
    force_y: Vec<f32>,
    dead: Vec<bool>,
    entity_dirty: Vec<u32>,  // only touched indices

    // Per-settlement economy (indexed by settlement ID)
    production: Vec<[f32; NUM_COMMODITIES]>,
    consumption: Vec<[f32; NUM_COMMODITIES]>,
    treasury_adj: Vec<f32>,
    settlement_dirty: Vec<u32>,

    // Per-faction, per-region — same pattern
    faction_fields: Vec<f32>,
    region_fields: Vec<f32>,
    // ...
}

Merging a Damage { target_id: 42, amount: 15.0 } is a single indexed add: self.damage[42] += 15.0. No hashing. No allocation.

The dirty lists track which indices were touched, so clearing between ticks is O(touched) not O(capacity). With ~200 entities taking damage per tick out of 2,000 total, that's a 10x speedup on the clear path alone.

Result: the merge phase consistently runs in 5–50µs.

167 Systems

The compute phase dispatches 167 independent systems. Each reads the frozen WorldState and returns Vec<WorldDelta>. Here's a sample:

// src/world_sim/systems/mod.rs
pub mod agent_inner;        // NPC needs, emotions, memory
pub mod economy;            // production, consumption, pricing
pub mod faction_ai;         // military buildup, diplomatic stance
pub mod warfare;            // sieges, conquest, wall damage
pub mod adventuring;        // party formation, quest execution
pub mod monster_ecology;    // reproduction, migration, den formation
pub mod legends;            // legendary NPC detection
pub mod death_consequences; // funerals, inheritance, mourning
pub mod civil_war;          // internal faction tension → revolt
pub mod espionage;          // spy missions, information gathering
pub mod romance;            // relationship formation
pub mod bloodlines;         // family lineages
pub mod oaths;              // loyalty, vengeance, trade oaths
pub mod religion;           // divine favor, temples, rituals
pub mod smuggling;          // black market trade routes
pub mod prophecy;           // visions and foreshadowing
// ... 150+ more

Systems break into two scheduling categories:

  • Settlement-scoped systems (economy, food, population, buildings, battles, etc.) run in parallel across settlements via rayon. Two different settlements never share mutable state, so these parallelize cleanly.
  • Global systems (seasons, weather, diplomacy, faction AI, quest lifecycle) must see the whole world and run sequentially.

A run_system! macro wraps each call with optional profiling:

macro_rules! run_system {
    ($name:expr, $func:expr, $state:expr, $out:expr) => {{
        #[cfg(feature = "profile-systems")]
        let _t = Instant::now();
        $func($state, $out);
        // ...
    }};
}

Fidelity Levels: Simulating What Matters

Not every entity needs the same resolution. A merchant traveling between cities doesn't need per-ability combat ticks. A settlement the player has never visited doesn't need per-NPC movement.

Four fidelity levels control how much work the sim does per entity:

LevelResolutionWhat Runs
High100ms tacticalPer-ability, per-tick combat
MediumSettlementMovement, production, work, trade
LowOverworldCoarse movement, encounter detection
BackgroundOff-screenAggregate stats only, zero per-entity deltas

Fidelity is location-based, not entity-based. A grid escalates to High when both hostile and friendly entities are present — combat is happening. It drops back to Medium when one side is eliminated.

The practical effect: during a siege, the besieged settlement runs at full tactical resolution while distant settlements tick at Background. The sim stays at 140–200 ticks/sec with 2,000 entities because most of them aren't doing expensive work.

NPC Inner Life

The agent inner state system (agent_inner.rs, ~1,500 lines) is where NPCs stop being stat blocks and start being characters.

Needs

Six needs drift and spike based on events:

NeedDrift SourceEvent Spikes
HungerTime (always rising)Eating (drops to 0)
SafetySettlement threat levelBattles, sieges, monster proximity
ShelterHas home building?Eviction, building destruction
SocialIsolation durationConversations, festivals, funerals
PurposeHas active goal?Quest completion, class progression
EsteemReputation, levelChronicle mentions, legendary deeds

Emotions

Six emotions spike from events and decay at ~0.005 per 100 ticks:

  • Joy — quest success, festival attendance, romance
  • Anger — friend death (if killer known), betrayal, theft
  • Fear — monster proximity, siege, low safety need
  • Grief — friend death, settlement destruction, exile
  • Pride — chronicle mention, legendary status, oath fulfillment
  • Anxiety — low purpose, threatening world events, debt

Memory & Beliefs

Each NPC has a 20-event ring buffer and a list of semantic beliefs with decaying confidence:

NPC sees friend die in battle
  → MemEvent { kind: WitnessedDeath, tick: 4200, entity_id: 17 }
  → Belief { content: "the northern frontier is dangerous", confidence: 0.8 }
  → Emotion spike: grief +0.6, anger +0.3 (if killer known)
  → Grudge formed against killer (decays at 0.001/100 ticks)

Beliefs decay over time but influence decisions. An NPC who believes a route is dangerous will avoid it even after the threat is gone — until the belief fades or a contradicting event overwrites it.

Personality

Personality is experience-shaped, not assigned at birth. An NPC who survives three sieges develops high risk_tolerance. One who mentors many apprentices develops high compassion. The personality then modifies decision weights: a high-ambition NPC prioritizes purpose over safety; a high-compassion NPC prioritizes social bonds.

WorldView Optimization

The agent loop pre-computes a WorldView snapshot before iterating over entities:

struct WorldView {
    threat: Vec<f32>,                        // per-settlement
    population: Vec<u32>,                     // per-settlement
    faction_stance: Vec<DiplomaticStance>,    // per-faction
    recent_deaths: Vec<u32>,                  // entity IDs
    grid_hostile_count: Vec<u32>,
    coworkers: HashMap<u32, Vec<u32>>,
    building_owner: HashMap<u32, u32>,
    resources: Vec<(u32, (f32, f32), ResourceType)>,
    npc_actions: HashMap<u32, (u8, f32)>,    // theory of mind
    // ...
}

Every NPC lookup during the agent loop is O(1) by ID — no scanning, no filtering. The WorldView is built once every 10 ticks, not every tick.

Tags, Classes, and Emergent Roles

NPCs don't choose a class at character creation. They become their class through behavior.

Compile-Time Tag Hashing

Every action in the sim emits behavior tags — stack-allocated bundles of up to 8 (tag, weight) pairs:

pub const fn tag(name: &[u8]) -> u32 {
    let mut hash: u32 = 2166136261;  // FNV-1a
    let mut i = 0;
    while i < name.len() {
        hash ^= name[i] as u32;
        hash = hash.wrapping_mul(16777619);
        i += 1;
    }
    hash
}

pub const MELEE: u32    = tag(b"melee");
pub const SMITHING: u32 = tag(b"smithing");
pub const FAITH: u32    = tag(b"faith");
// 35+ tags total

Tags are FNV-1a hashed at compile time — zero runtime cost to create, and comparing two tags is a single u32 ==.

When an NPC mines ore, they accumulate MINING and ENDURANCE tags. When they fight, they get COMBAT and MELEE or RANGED. When they trade between settlements, TRADE and NAVIGATION.

Class Matching

Every 50 ticks, the system compares each NPC's accumulated behavior profile against 12 class templates:

ClassKey Tags
Warriorcombat, melee, endurance
Rangerranged, navigation, survival
Merchanttrade, negotiation, navigation
Paladincombat, faith, discipline
Roguestealth, deception, awareness
Healermedicine, herbalism, compassion
Scholarresearch, lore, teaching

NPCs can hold multiple classes simultaneously, each with independent level tracking. A level increments every 50 ticks if the behavior tags still match. An NPC who fought for 200 ticks then switched to smithing full-time would be a Warrior 4 / Blacksmith 2.

If no template matches but behavior tags are significant, the system generates a procedural class — unique to that NPC, named from their dominant tag combinations.

Ability Generation

When an NPC gains a class, the ability generator synthesizes abilities using grammar-driven DSL generation:

  • Walk the ability DSL tree with archetype-conditioned distributions (knight → physical damage, mage → area effects)
  • Name abilities thematically: archetype prefix (iron/wild/arcane) + effect suffix (strike/mend/ward)
  • Score quality to prevent degenerate combinations
  • Rate-limited to 10 abilities per class-assignment pass

The generated abilities are real — the same DSL format used by hand-authored hero templates, runnable in the tactical combat sim.

Economy

Eight commodities flow through the settlement network:

FOOD · IRON · WOOD · HERBS · HIDE · CRYSTAL · EQUIPMENT · MEDICINE

Each settlement has per-commodity stockpiles, production rates, and consumption rates. Prices emerge from supply and demand — the price_discovery system adjusts prices based on stockpile levels relative to consumption.

NPCs make economic decisions: a miner in a settlement with excess IRON and scarce FOOD might switch to farming. Merchants spot price differentials between settlements and run trade routes, buying low and selling high. The trade_routes system tracks profitable routes, and NPCs who complete profitable trades accumulate TRADE tags — eventually becoming Merchant-class.

Known tech debt: gold can appear from thin air (treasure hunts, bounties, dungeon loot). There's no rent or housing costs yet. This causes slow inflation over long runs.

Warfare and Sieges

The warfare system models multi-phase sieges:

  1. Approach (50 ticks): monster forces converge on a walled settlement
  2. Assault (100 ticks): attackers damage walls, defenders fight from watchtowers (+range)
  3. Breach: if walls fall, attackers enter the settlement grid — fidelity escalates to High, and the tactical combat sim takes over

The trigger: 100+ hostile monsters near a walled settlement with 10+ within 50 distance units.

Survivors of sieges gain massive RESILIENCE and COMBAT tags. Siege survivors who were already legendary get chronicle entries that echo through the world — other NPCs hear about the defense, morale shifts, grudges form against the attacking faction.

Legends and Death

NPCs become legendary through deeds, not levels. The criteria:

  • 5+ chronicle mentions
  • 2+ distinct classes
  • Survived 3+ friend deaths

Legends boost settlement morale (+5 for all residents), attract followers seeking purpose, and get "the Legendary" appended to their name.

When a legend dies, the consequences cascade:

  • Worldwide mourning: all NPCs with a relationship take grief spikes
  • Funeral: 20-tick ceremony at the settlement
  • Legacy: top apprentice inherits 30% of the mentor's behavior tags
  • Grudge formation: witnesses who saw the killer form vengeance grudges (decay 5x slower than normal beliefs)
  • Chronicle entry: rate-limited to 5 per tick, gated to level 30+ for notable deaths

The apprentice lineage system means skills propagate through generations. A legendary smith's apprentice carries forward smithing tags, and if they reach legendary status themselves, the chronicle tracks the unbroken lineage.

Performance

Measured on a typical run with 2,000 entities and 8 settlements:

PhaseTime
Compute (167 systems)140–180µs
Merge (flat-array fold)5–50µs
Apply (next state)50–100µs
Post-apply (agent inner, pathfinding, etc.)50–150µs
Total per tick~300–500µs

Key performance decisions:

  • Dirty-list clearing: O(touched) not O(capacity) — with 200 entities taking damage out of 2,000, 10x faster clear
  • WorldView pre-computation: built once every 10 ticks, not every tick
  • Entity compaction: dead entities removed from the array every 500 ticks, not immediately
  • Class matching: every 50 ticks, not every tick
  • Fidelity scaling: Background entities produce zero per-entity deltas

What Emerges

None of the following behaviors are scripted. They emerge from system interactions:

An NPC witnesses their mentor's murder → forms a grudge → joins an adventuring party biased toward the killer's region → finds and kills the murderer → pride spike → gains "Avenger" chronicle entry → becomes legendary

A faction goes hostile → military buildup → siege on a border settlement → wall breach → tactical combat at High fidelity → survivors gain resilience tags → NPC with enough combat + faith tags becomes a Paladin → attracts followers → settlement morale recovers

A merchant discovers a price differential → runs a trade route → accumulates trade tags → becomes Merchant-class → gains ability "Iron Negotiation" → their apprentice inherits trade tags → continues the route after the mentor retires

A settlement's food production drops → hunger need spikes → NPCs migrate to better settlements → depopulated settlement produces refugees with trauma beliefs → refugees spread rumors about the failing settlement → neighboring settlements increase food production in response

The simulation doesn't know about "stories." It knows about needs, deltas, tags, and thresholds. Stories are what happens when 167 systems talk to each other through 2,000 NPCs over 10,000 ticks.
